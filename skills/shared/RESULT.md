# Skill Result Interface

Standard format for result files written by skills for the `/project` orchestrator.

## Purpose

Result files provide the orchestrator with:
1. **Completion status** - success or failure
2. **Outputs** - files modified
3. **Decisions** - choices made with context, reason, and alternatives
4. **Learnings** - insights captured during execution
5. **Escalations** - questions that need user resolution
6. **Context for next** - curated summary for subsequent skills

## Required Schema (Minimal)

The control loop requires this minimal structure:

```toml
# Required sections
[status]
success = true  # boolean: did the skill complete successfully?

[outputs]
files_modified = ["docs/requirements.md", "docs/design.md"]

# Optional sections
[[decisions]]
context = "Analyzing CLI structure"
choice = "Use subcommands instead of flags"
reason = "More intuitive for users"
alternatives = ["Flat flags", "Positional arguments"]

[[learnings]]
content = "The codebase uses targ for CLI parsing"
```

### Validation Rules

- `[status]` section is **required** with `success` boolean
- `[outputs]` section is **required** with `files_modified` list
- `[[decisions]]` is optional, but if present must have `context`, `choice`, `reason`
- `[[learnings]]` is optional, but if present must have `content`
- `alternatives` in decisions is optional

### Invalid Examples

**Missing status:**
```toml
# INVALID - no [status] section
[outputs]
files_modified = []
```

**Missing decision fields:**
```toml
# INVALID - decision missing context and reason
[[decisions]]
choice = "Use subcommands"
```

**Empty learning:**
```toml
# INVALID - learning missing content
[[learnings]]
# no content field
```

## Extended Format (Full)

For richer orchestrator features, skills can provide additional metadata:

## File Format

Result files use TOML format for structured data with clear types.

### Location

```
<project-dir>/.claude/context/<skill-name>-result.toml
```

Example: `.claude/context/pm-infer-result.toml`

### Standard Structure

```toml
# Skill Result File
# Generated by skill at completion

[result]
skill = "pm-infer"           # Skill that produced this result
status = "success"           # "success", "failure", "needs_escalation"
timestamp = 2024-01-15T11:45:00Z
duration_seconds = 180

[result.summary]
# Human-readable summary of what was accomplished
text = "Inferred 12 requirements from codebase analysis"

[outputs]
# Artifacts generated or modified

[outputs.requirements]
path = "docs/requirements.md"
action = "created"           # "created", "modified", "unchanged"
items_count = 12
id_range = "REQ-001 to REQ-012"

[outputs.traceability]
# Traceability is inline via **Traces to:** fields in artifacts
traces_added = 15  # Number of **Traces to:** fields added/updated

[[escalations]]
# Questions that need user resolution

[[escalations]]
id = "ESC-001"
category = "requirement"
context = "Analyzing CLI flags"
question = "Should --verbose be a global flag or per-command?"
suggested_answer = "Global flag is more consistent with other CLI tools"

[[escalations]]
id = "ESC-002"
category = "requirement"
context = "Analyzing authentication"
question = "Is OAuth support required, or only API key auth?"

[context_for_next]
# Curated summary for subsequent skills
# Should be concise but complete enough to inform next skill

[context_for_next.artifacts]
requirements_summary = """
12 requirements identified:
- REQ-001 to REQ-004: Core CLI functionality
- REQ-005 to REQ-008: Argument parsing
- REQ-009 to REQ-012: Help and documentation
"""

[context_for_next.decisions]
# Key decisions made that affect subsequent phases
decisions = [
    "Using structured flag parsing over positional arguments",
    "Help text generated from struct tags",
]

[context_for_next.open_questions]
# Questions passed to next skill or collected for escalation
questions = [
    "ESC-001: Global vs per-command verbose flag",
    "ESC-002: OAuth vs API key authentication",
]

[errors]
# Only present if status = "failure"
message = "Failed to parse existing requirements.md"
details = "Line 45: Invalid YAML frontmatter"
recoverable = true
```

## Status Values

| Status | Meaning | Orchestrator Action |
|--------|---------|---------------------|
| `success` | Skill completed successfully | Continue to next phase |
| `failure` | Skill failed, cannot continue | Report error, offer retry |
| `needs_escalation` | Success but has escalations | Collect escalations, continue |

## Escalation Format

Each escalation must have:
- `id`: Unique identifier (ESC-NNN)
- `category`: requirement, design, architecture, task
- `context`: What was being analyzed
- `question`: Clear question needing resolution
- `suggested_answer`: Optional recommended answer

## Context for Next

The `context_for_next` section is critical for maintaining continuity:

1. **artifacts**: Summary of generated artifacts
2. **decisions**: Key decisions that affect subsequent work
3. **open_questions**: Unresolved questions to carry forward

The orchestrator uses this to write the next skill's context file without re-reading all artifacts.

## Writing Results in Skills

Skills should write results at completion:

```markdown
## On Completion

1. Gather outputs:
   - List all files created/modified
   - Count items generated

2. Format escalations:
   - Assign ESC-NNN IDs
   - Include context and question

3. Prepare context_for_next:
   - Summarize artifacts (don't include full content)
   - List key decisions
   - Carry forward open questions

4. Write result file:
   ```bash
   projctl context write --dir <project-dir> --skill <skill-name> --result --file <result.toml>
   ```
```

## Example: pm-infer Result

```toml
[result]
skill = "pm-infer"
status = "needs_escalation"
timestamp = 2024-01-15T11:45:00Z
duration_seconds = 180

[result.summary]
text = "Inferred 12 requirements. 2 escalations need resolution."

[outputs.requirements]
path = "docs/requirements.md"
action = "created"
items_count = 12
id_range = "REQ-001 to REQ-012"

[[escalations]]
id = "ESC-001"
category = "requirement"
context = "Analyzing CLI flags"
question = "Should --verbose be a global flag or per-command?"
suggested_answer = "Global flag for consistency"

[[escalations]]
id = "ESC-002"
category = "requirement"
context = "Analyzing authentication"
question = "Is OAuth support required?"

[context_for_next]

[context_for_next.artifacts]
requirements_summary = """
12 requirements (REQ-001 to REQ-012):
- Core CLI: 4 requirements
- Argument parsing: 4 requirements
- Help/docs: 4 requirements
"""

[context_for_next.decisions]
decisions = [
    "Structured flag parsing preferred",
    "Help from struct tags",
]

[context_for_next.open_questions]
questions = ["ESC-001", "ESC-002"]
```

## Validation

Result files are validated by `projctl context read --result`:
- Required: `result.skill`, `result.status`, `result.timestamp`
- `escalations` must have `id`, `category`, `question`
- `outputs` entries must have `path` and `action`
