# Memory Tier System: Research Synthesis and Optimization Framework

**Date:** 2026-02-13
**Research Team:** skills-researcher, claudemd-researcher, embeddings-researcher, hooks-researcher, audit-agent
**Synthesis Agent:** synthesis-agent

---

## Executive Summary

The memory tier system is **over-engineered on tier movement machinery** (promote/demote/merge/split plumbing with 12 configurable thresholds) while **under-investing in content quality optimization** at each tier. Research across all five dimensions (skills, CLAUDE.md, embeddings, hooks, current implementation) converges on the same insight:

> **Content quality matters more than mechanical sophistication.**

The system has sophisticated plumbing that reliably moves noise between tiers. The synthesis produces "important pattern for review" 56+ times in Promoted Learnings—a clear signal that we've optimized mechanics before validating that basic quality operations work.

**Core Finding:** The tier question is not "where does content live?" but rather "how strongly do we enforce quality?" This reframes the architecture as:

1. **Hooks** (strongest enforcement) → Deterministic quality gates
2. **CLAUDE.md** (always-loaded guidance) → Judgment calls and routing
3. **Skills** (on-demand procedural knowledge) → Domain-specific workflows
4. **Embeddings** (episodic retrieval) → Historical learnings and patterns

**Recommended Shift:** From "sophisticated tier movement" to "quality at each tier + deterministic enforcement."

## Section 1: What Each Tier Should Optimize For

### 1.1 Embeddings (Tier 3): Episodic Memory & Retrieval

**Purpose:** Store and retrieve historical learnings, patterns, and context that aren't universal enough for CLAUDE.md or structured enough for skills.

**Optimal Content Characteristics:**

**Length:**
- **Current:** 128 max sequence length (below model's 512 token capacity)
- **Ideal:** 512 tokens per chunk—full model capacity for semantic coherence
- **Rationale:** Longer chunks = better retrieval quality (research shows 87% accuracy vs 50% baseline for adaptive chunking). Only retrieved when relevant, so no always-loaded cost penalty.

**Structure:**
- **Semantic enrichment** (CURRENT STRENGTH ✅): LLM extraction creates:
  - observation_type, concepts, principle, anti_pattern, rationale
  - enriched_content combines into semantically rich embedding text
  - This is industry best practice—dramatic retrieval quality improvement
- **Single-topic focus:** Memories are typically single-topic learnings (no chunking needed)
- **Rich metadata:** confidence, retrieval_count, projects_retrieved, timestamps, source_type

**Retrieval Characteristics:**

**What's Working:**
- ✅ Hybrid search (BM25 + vector similarity + Reciprocal Rank Fusion)
- ✅ Confidence weighting prevents low-quality matches from dominating
- ✅ Retrieval tracking identifies frequently-used vs rarely-used content
- ✅ Deduplication at learn-time (0.98 threshold) with LLM-driven decisions

**Critical Gap:**
- ❌ **Missing similarity threshold filtering**—system always returns top N results even if all are poor matches
- ❌ **Recommendation:** Add query-time threshold (default 0.7-0.8 cosine similarity)
- ❌ **Surface fewer, higher-quality retrievals** to reduce context window competition

**Content Quality Markers:**
- High semantic relevance to query (similarity > 0.7-0.8)
- Actionable guidance (principles, anti-patterns, concrete examples)
- Retrieved from similar project contexts
- Reinforced through repeated use (high retrieval_count)
- Specific and clear (not generic platitudes)

**Model Selection:**
- **Embedding generation:** Haiku (high-volume, well-defined pattern matching, 3x cost savings)
- **Similarity search:** Haiku (speed critical for retrieval)
- **Semantic analysis of results:** Sonnet (quality judgment of retrieved content)

**Model-Specific Optimization:**
- **If using e5-small-v2:** Add "query:" prefix to search text, "passage:" prefix to stored memories (CRITICAL—not currently implemented)
- **If using all-MiniLM-L6-v2:** Current approach is fine (no prefix needed)
- **Current gap:** Code downloads all-MiniLM-L6-v2 despite variable names suggesting e5—clarify which model is actually used

**Optimization Priority:** Content quality (semantic enrichment, actionable principles) > retrieval precision (threshold filtering) > mechanical sophistication (tier movement).

### 1.2 Skills (Tier 2): Procedural Knowledge & Workflows

**Purpose:** On-demand procedural knowledge—teach HOW to do something. Loaded when semantically relevant to current task.

**Optimal Content Characteristics:**

**Length:**
- **Current:** ~2000 tokens per SKILL.md (reasonable)
- **Discovery metadata:** Currently 128 tokens, model supports 512
- **Ideal:** Richer descriptions (256-384 tokens) structured for discovery:
  ```
  Core purpose (20 tokens) + Trigger contexts (40 tokens) +
  File types/domains (30 tokens) + When NOT to use (30 tokens) +
  Related concepts/keywords (50 tokens)
  ```
- **Rationale:** Description is ALWAYS loaded (unlike full SKILL.md). Must be rich enough to distinguish between 25+ skills. Community best practice: include what/when/trigger contexts/file types/keywords.

**Structure:**
- **Progressive disclosure:** metadata (~100 tokens) → full instructions (<5k tokens) → resources (on-demand)
- **Bounded scope:** ONE thing well, no orchestration
- **Contract-driven:** Explicit inputs/outputs/checks (especially for producers)
- **Crisp semantic descriptions:** Gerund form naming (e.g., "Producing requirements from user interviews")
- **Structured workflow patterns:** GATHER→SYNTHESIZE→PRODUCE (clear phases)
- **Optional REFERENCE.md:** Supplemental docs for pattern libraries

**Discovery Patterns:**
- **Primary mechanism:** YAML frontmatter description (semantic embedding-based)
- **Two modes:** User `/slash` invocation OR auto-discovery when description matches task
- **Organizational patterns:**
  - Role-based grouping (producer, qa, evaluator)
  - Phase-based grouping (tdd-red, tdd-green, tdd-refactor)
  - Naming conventions (suffix: `-producer`, `-infer`, `-interview`)

**Scale Threshold:**
- **Under 30 skills:** No organization needed if descriptions are crisp
- **30-100 skills:** Use naming conventions and role fields
- **100+ skills:** Consider namespacing or sub-categories
- **Current state:** 21 skills with clear role/phase organization = healthy
- **Key insight:** Semantic discovery scales—"skills with poor descriptions cause discovery failures, not quantity"

**Content Quality Markers:**
- **Context efficiency:** Only add what Claude doesn't already have (avoid generic advice)
- **Actionable workflows:** Clear steps, not vague principles
- **Domain specificity:** D3.js, Playwright, SRE-specific patterns (not generic coding advice)
- **Bounded scope:** If skill needs sub-skills, it's too complex—split it
- **Testable contracts:** Outputs and checks are verifiable

**Model Selection:**
- **Skill creation/editing:** Sonnet (procedural knowledge requires nuance)
- **Skill retrieval/scoring:** Haiku (fast similarity matching for discovery)
- **Skill refactoring:** Sonnet (understand contracts and dependencies)

**Skills vs Agents Boundary:**
- **Skills teach expertise** (reusable knowledge, shares context, portable)
- **Agents execute independently** (isolated context, different tools/permissions)
- **Rule:** Skills teach HOW; agents DO the work in isolation

**Optimization Priority:** Discoverable descriptions > contract clarity > workflow structure > mechanical generation.

### 1.3 CLAUDE.md (Tier 1): Universal Guidance & Routing

**Purpose:** Always-loaded guidance for judgment calls and routing to more specific knowledge. Highest context cost—every line competes for attention.

**Optimal Content Characteristics:**

**Length:**
- **Industry standard:** <200 lines ideal, <300 lines max
- **Current:** 259 lines (over target)
- **This project target:** <100 lines (aggressive but achievable)
- **Core constraint:** Context window fills fast, performance degrades—every line competes for attention

**Content Hierarchy (Priority Order):**
1. **Pointers/references** → "Use /project for feature work" (routing)
2. **Specific rules** → "Trailer is AI-Used: [claude]" (exact format)
3. **Decision trees** → "When to use EnterPlanMode" (branching logic)
4. **Critical warnings** → Things Claude repeatedly violates
5. **Checklists** → Pre-flight checks (if deterministic, move to hooks)
6. **Principles** → High-level behavioral guidelines (lowest priority)

**Anti-Patterns (Should NOT be in CLAUDE.md):**
- ❌ Generic advice ("write good code")
- ❌ Code snippets (put in skills or documentation)
- ❌ Domain workflows (TDD process → tdd-* skills)
- ❌ Vague directives ("be careful")
- ❌ Content better enforced by hooks (quality gates, format validation)

**Migration Candidates (CLAUDE.md → Skills):**
- "Code & Debugging" section → tdd-* skills
- "Planning & Execution" section → project/plan-producer skills
- "Documentation" section → doc-producer skill
- "Traceability" section → alignment-producer skill
- "Audits" section → qa/audit skills

**Migration Candidates (CLAUDE.md → Hooks):**
- Git commit trailer enforcement → Pre-commit hook
- No amend on pushed commits → Pre-commit hook that checks remote status
- Quality gate enforcement (targ check) → Pre-commit hook
- Traceability validation (projctl trace validate) → Pre-commit hook
- Test execution → Pre-commit hook ensuring tests pass

**What Stays in CLAUDE.md:**
- Tier routing logic ("Choose the Right Tier")
- Meta-routing ("Use Skills First")
- Critical warnings (repeatedly violated rules)
- Behavioral guidelines (judgment calls)
- **Pointers to where details live** (not the details themselves)

**Three-Level Enforcement Hierarchy:**
1. **Hooks** (strongest) → Deterministic enforcement, always runs
2. **CLAUDE.md** (medium) → Always-loaded guidance for judgment calls
3. **Skills** (weakest but most flexible) → On-demand procedural knowledge

**Decision Rule for Content:**
- **Deterministic & always applies** → Hook
- **Requires judgment or context-dependent** → CLAUDE.md
- **Domain-specific or procedural** → Skill
- **Current CLAUDE.md has too much middle-ground content** that should move to extremes

**Model Selection:**
- **Content edits:** Sonnet (balance quality and speed)
- **Promotion decisions (Skills → CLAUDE.md):** Opus (cross-context reasoning required)
- **Demotion decisions (CLAUDE.md → Skills):** Sonnet (clear domain specificity signal)
- **Major restructuring:** Opus (high stakes, affects all sessions)

**Optimization Priority:** Brevity + pointers > enforcement via hooks > content migration to skills.

### 1.4 Hooks: Deterministic Enforcement Layer

**Purpose:** Enforce quality gates and behavioral requirements without relying on model memory. Highest enforcement strength—violations block operations.

**Hook Types & Use Cases:**

**PreToolUse** - Before tool execution
- Validate dangerous operations (no git checkout --, no force push to main)
- Secret scanning before git commit
- File permission checks before writes

**PostToolUse** - After tool completion
- Auto-formatting (Prettier, Black) after code edits
- Linting after file changes
- Triggering builds or tests

**Stop** - End-of-turn validation
- CLAUDE.md size enforcement (<100 lines)
- Quality gates (targ check, linting, type checks)
- Repository-wide guardrails

**TeammateIdle** - Before agent goes idle
- Skill contract validation (required sections, token count, no TODOs)
- Output file verification
- Test coverage thresholds
- **Critical:** Prevents premature idle when quality standards unmet

**Exit Code Semantics:**
- **Exit 0:** Success, continue normally
- **Exit 2:** BLOCKING error—Claude receives stderr as feedback and adjusts

**Top 3 Quality Gates for Memory Tier System:**

1. **CLAUDE.md Size Enforcement (Stop Hook)**
   - Validates <100 line budget at end-of-turn
   - Exit 2 if over budget with specific count
   - Prevents context bloat at most expensive tier

2. **Skill Contract Validation (TeammateIdle Hook)**
   - Validates SKILL.md structure before agent completes
   - Required: Contract, Examples, Traceability sections
   - Token count within ~2000 budget
   - No TODO/FIXME comments (incomplete work)
   - Ensures tier 2 quality before persistence

3. **Embedding Metadata Completeness (PostToolUse Hook)**
   - Validates embedding metadata when created/updated
   - Confidence score in valid range [0.0-1.0]
   - Context field populated
   - At least one tag for clustering
   - Maintains tier 3 retrieval effectiveness

**Hook vs CLAUDE.md Decision Rule:**
- **Hook:** Deterministic, always applies, can be automated
- **CLAUDE.md:** Requires judgment, context-dependent, advisory
- **Example:** "No blanket lint overrides" requires judgment → stay in CLAUDE.md

**Current State:**
- ❌ **No hooks configured** (only permissions defined in .claude/settings.local.json)
- ❌ **Major opportunity:** 5-7 CLAUDE.md rules should be hooks
- ❌ **Impact:** Migrating to hooks would remove ~20-30 lines from CLAUDE.md AND improve enforcement reliability

**Model Selection for Hook Operations:**
- Hooks don't use models directly (shell commands)
- But agents responding to hook feedback use Sonnet (understand error, adjust approach)

**Optimization Priority:** Implement top 3 quality gates > migrate CLAUDE.md enforcement rules > expand to additional validation.

### 1.5 Agent Files: Isolated Execution Context

**Purpose:** Independent agents with different tools/permissions for isolated execution. NOT part of the tier hierarchy—orthogonal to memory system.

**When to Use:**
- **Independent execution:** Task requires isolation from main context
- **Different tools/permissions:** Subagent needs restricted tool access
- **Parallel work:** Multiple agents working concurrently on separate tasks
- **Specialized roles:** QA, research, exploration with focused capabilities

**Agent Files vs Skills:**
- **Skills:** Teach expertise within shared context (memory tier 2)
- **Agents:** Execute independently with isolated context (not a memory tier)
- **Don't confuse:** Agent files are NOT "agent-generated skills" or a tier

**Model Selection for Agents:**
- **Exploration agents:** Haiku (fast scanning, pattern matching)
- **Implementation agents:** Sonnet (code generation, bug fixing)
- **Strategic coordination (team lead):** Opus (multi-agent orchestration)
- **QA agents:** Sonnet (judgment of quality)

**Integration with Memory Tiers:**
- Agents can READ from all memory tiers (embeddings, skills, CLAUDE.md)
- Agents can WRITE to memory tiers (create skills, add embeddings, propose CLAUDE.md changes)
- TeammateIdle hooks can enforce quality before agents complete

### 2.2 Where We're Over-Engineering Mechanics

**1. Triple Merge/Split Code Paths**
- optimizeMergeSkills() (active skills)
- Merge during optimizeCompileSkills() (check existing by slug, update if found)
- Periodic reorganization merge (re-cluster everything, consolidate)
- **Problem:** Three separate implementations of conceptually the same operation

**2. Complex Promotion/Demotion Chains**
Each tier transition has 4-6 steps:
- Embeddings → Skills: cluster detection → scoring → theme generation → LLM content → file write → DB insert
- Skills → CLAUDE.md: utility check → project count → LLM synthesis → semantic dedup → file append → DB update
- CLAUDE.md → Skills: specificity detection → skill generation → file write → file removal
- **Problem:** User value unclear for many transitions—do they actually help?

**3. Threshold Tuning Complexity**
12 parameters:
- DecayBase, PruneThreshold, DupThreshold, SynthThreshold
- MinClusterSize, MinRetrievals, MinProjects
- MinSkillUtility, MinSkillConfidence, MinSkillProjects, AutoDemoteUtility, ReorgThreshold
- **Problem:** Interdependencies not documented, no guidance on when to adjust, no measurement of impact

**4. Periodic Reorganization Overhead**
Every 30+ days:
1. Fetch ALL memories with embeddings
2. Re-cluster at ReorgThreshold (0.8)
3. Regenerate all skill content, themes, descriptions
4. Update existing skills OR create new ones
5. Prune orphaned skills
- **Problem:** Expensive (O(n²) similarity matrix), disruptive (all skills regenerated), unvalidated (no before/after quality measurement)

**5. State Tracking Proliferation**
Embeddings table: 14+ columns (promoted, promoted_at, confidence, retrieval_count, projects_retrieved, observation_type, memory_type, principle, rationale, concepts, anti_pattern, enriched_content, observation_facts, source, source_type, created_at, last_retrieved)
Skills table: 15+ columns (alpha, beta, utility, retrieval_count, last_retrieved, pruned, claude_md_promoted, promoted_at, merge_source_ids, split_from_id, demoted_from_claudemd, source_memory_ids, created_at, updated_at, embedding_id)
- **Problem:** Many columns overlap in purpose or go unused

### 2.3 Where We're Under-Investing in Content Quality

**1. Synthesis Produces Noise, Not Knowledge**
- Current generatePattern(): clusters by >0.8 similarity (good), extracts top-3 keywords via word frequency (weak)
- **Result:** "important pattern for review" label appears **56+ times** in Promoted Learnings
- LLM synthesis exists (GeneratePatternLLM) but:
  - Output quality not validated
  - No check for actionability vs description
  - No verification of non-redundancy with existing CLAUDE.md

**2. Promotion Criteria Are Mechanical, Not Qualitative**
Embeddings → CLAUDE.md gate:
- retrieval_count >= 5 ✓
- confidence >= 0.8 ✓
- unique_projects >= 3 ✓
- principle != '' ✓ (ISSUE-210 enrichment gate)

**Missing checks:**
- ❌ Is this principle **actionable**? ("Always X" vs "X is important")
- ❌ Is this **specific**? (Vague wisdom vs concrete rule)
- ❌ Is this **clear**? (Will I understand this 6 months from now?)
- ❌ Is this **non-redundant** with existing CLAUDE.md entries?
- **Problem:** Frequency ≠ quality

**3. Missing Content Refinement Operations**
No operations to improve existing content:
- **Rewrite:** "Make this principle clearer"
- **Merge with context:** "Combine these 3 similar principles while preserving nuance"
- **Extract examples:** "Separate principle from implementation examples"
- **Add rationale:** "Enrich with *why* this matters, not just what"
- **Current operations:** Only add (promote), remove (demote/prune), consolidate (pick longer string)

**4. Split Operations Are Stubs**
- Embeddings split: returns fmt.Errorf("not yet implemented")
- Skills split: re-clusters at 0.6 but no topic detection—just splits by similarity, not semantic coherence
- CLAUDE.md split: uses sentence boundaries and conjunctions—syntactic, not semantic
- **Problem:** No LLM-based topic detection for any tier

**5. Consolidate Is Merge-Only, Loses Nuance**
Current mergeEntries():
```go
if len(entry1) > len(entry2) {
    return entry1
}
return entry2
```
- Picks longer string, discards shorter one entirely
- **Missing:** LLM-driven consolidation that captures insights from both entries
- **Problem:** Loses context, nuance, and potentially critical distinctions

**6. User Value Unclear for Many Operations**
Unanswered questions:
- Do dynamically generated skills actually improve retrieval vs just adding complexity?
- When does 30-day re-clustering improve quality vs just churn the skill set?
- What's the user experience improvement from splitting a skill?
- When does merging improve clarity vs lose important distinctions?
- **Problem:** No metrics for retrieval precision, CLAUDE.md usability, skill effectiveness

## Section 3: Model Selection Strategy

### 3.1 When Opus/Sonnet/Haiku Each Shine

**Claude Opus 4.5/4.6** - Highest Capability
- **Benchmarks:** 80.9% on SWE-bench Verified (resolves ~8/10 real bugs)
- **Strengths:** Architectural reasoning, multi-file refactoring with deep context, complex design/research
- **Speed:** Moderate (longer responses, deeper analysis)
- **Cost:** Premium (most expensive)
- **Use cases:**
  - Strategic decisions requiring cross-context understanding
  - Multi-file refactoring where context dependencies matter
  - Complex research requiring synthesis across many sources
  - Safety net before merging or releasing (final validation)

**Claude Sonnet 4.5** - Balanced (RECOMMENDED DEFAULT)
- **Benchmarks:** Best balance of capability and cost
- **Strengths:** Daily coding, bug fixes, debugging, tests, documentation, code reviews, most refactoring
- **Speed:** Fast (typically seconds)
- **Cost:** Mid-tier (5x Haiku cost for ~7.6% accuracy gain over Haiku)
- **Use cases:**
  - Standard development work
  - Writing tests and documentation
  - Code reviews and quality analysis
  - Most tier 2 operations (skill creation/editing)
  - Tier 1 operations (CLAUDE.md edits, demotion decisions)
- **Anthropic recommendation:** "Start with Sonnet 4.5 if unsure"

**Claude Haiku 4.5** - Speed and Efficiency
- **Benchmarks:** 73.3% on SWE-bench Verified, 90% of Sonnet's agentic coding capability
- **Strengths:** Real-time/high-volume tasks, initial queries, data classification, pattern matching
- **Speed:** Fastest (near-instant responses, 2x Sonnet speed)
- **Cost:** Most affordable (1/3 of Sonnet = 3x cost savings)
- **Use cases:**
  - Pattern completion when solution space is well-defined
  - High-frequency operations (embedding generation, similarity search)
  - Simple validations (format checks, structure validation)
  - Initial exploration before escalating to Sonnet

**Decision Heuristic:**
- **Complex design/research** → Opus
- **Everything else** → Sonnet (default)
- **High-frequency/well-defined tasks** → Haiku

### 3.2 How Model Selection Interacts with Each Tier

**Tier 3 (Embeddings) → Haiku-First Strategy**

| Operation | Model | Rationale |
|-----------|-------|-----------|
| Embedding generation | Haiku | High-volume (every memory), well-defined (pattern matching), 3x cost savings |
| Similarity search | Haiku | Speed critical for retrieval responsiveness |
| Semantic analysis of results | Sonnet | Quality judgment—which retrievals are actually useful? |
| Enrichment (principle extraction) | Sonnet | Nuanced understanding required for quality enrichment |

**Pattern:** Haiku for high-frequency operations (generation, search), Sonnet for quality analysis.

**Cost Impact:** Embeddings are most frequent tier operation—using Haiku here yields maximum savings.

**Rationale:** CLAUDE.md mistakes are expensive (always-loaded context). Promotion decisions require cross-context reasoning ("Does this belong in universal guidance?").

- **Agent responding to hook feedback:** Sonnet (understand error message, adjust approach)

## Section 4: Recommendations

### 4.1 What to Change About Promotion/Demotion Approach

**Current Approach:**
- Mechanical thresholds (retrieval_count, confidence, projects_retrieved)
- Focus on tier movement machinery (promote/demote/merge/split plumbing)
- Missing qualitative judgment (actionability, clarity, non-redundancy)

**Recommended Approach:**

**Shift from "Tier Movement" to "Content Quality + Enforcement"**

Instead of asking "Should this be promoted?", ask:
1. **"Is this universally useful?"** (belongs in always-loaded CLAUDE.md)
2. **"Is this procedural knowledge?"** (belongs in on-demand skills)
3. **"Is this deterministic enforcement?"** (belongs in hooks)
4. **"Is this episodic context?"** (stays in embeddings)

**Reframe the Architecture:**
Not 3 tiers (embeddings → skills → CLAUDE.md), but rather:
1. **Enforcement strength:** Hooks (strongest) > CLAUDE.md > Skills > Embeddings
2. **Context cost:** CLAUDE.md (always-loaded) > Skills (on-demand) > Embeddings (query-only)
3. **Content type:** Deterministic rules (hooks) > Routing/judgment (CLAUDE.md) > Procedures (skills) > Patterns (embeddings)

**Promotion Decision Framework:**

**Embeddings → Skills:**
- **When:** Cluster of 3+ similar memories forms a procedural pattern
- **Check:** Is this a reusable workflow? (not just a fact or observation)
- **Quality gate:** LLM synthesis produces structured workflow (GATHER→SYNTHESIZE→PRODUCE), not keyword label
- **Validation:** Before creating skill, verify:
  - Non-redundant with existing skills (similarity < 0.85)
  - Actionable (clear steps, not vague principles)
  - Bounded scope (ONE thing well)

**Skills → CLAUDE.md:**
- **When:** Skill is retrieved across 3+ projects AND utility >= 0.8
- **Check:** Is this universal guidance? (not domain-specific)
- **Quality gate (NEW):** LLM validates:
  - ✅ **Actionable:** "Always X" or "Never Y" (not "X is important")
  - ✅ **Specific:** Concrete rule (not vague wisdom)
  - ✅ **Clear:** Understandable 6 months from now
  - ✅ **Non-redundant:** Adds new information vs existing CLAUDE.md entries
- **Format:** Pointer to skill ("See /project for feature work"), not inline content

**CLAUDE.md → Hooks:**
- **When:** Rule is deterministic and always applies
- **Check:** Can this be automated? (no judgment required)
- **Examples:** Git trailer format, quality gates, traceability validation, test execution
- **Impact:** Removes ~20-30 lines from CLAUDE.md, improves enforcement reliability

**Demotion Decision Framework:**

**CLAUDE.md → Skills:**
- **When:** Content is domain-specific or procedural
- **Check:** Does this require context from the always-loaded tier, or can it be on-demand?
- **Examples:** TDD workflows, git patterns, documentation structure
- **Impact:** Reduces CLAUDE.md bloat, preserves knowledge in discoverable form

**Skills → Embeddings:**
- **When:** Skill unused (retrieval_count = 0 for 90+ days) OR utility < 0.4
- **Check:** Is this actually helping users, or just architectural complexity?
- **Action:** Don't delete—demote to embeddings (preserves learning, allows re-clustering)

**CLAUDE.md → Delete:**
- **When:** Content is demonstrably unused or outdated
- **Check:** User study or explicit feedback indicates this doesn't prevent mistakes
- **Rare:** Default to demotion, not deletion

**Stop Building Before Measuring:**
- Defer periodic reorganization (30-day re-clustering) until we can measure impact
- Defer dynamic skill generation (ISSUE-186) until we validate skills improve retrieval
- Defer background daemon (ISSUE-211) until we measure where 60s overhead matters
- **Build measurement first:** User feedback loop, retrieval precision metrics, CLAUDE.md effectiveness tracking

### 4.2 What to Change About Content Optimization

**Current Focus:** Tier movement machinery (promote/demote/merge/split)
**Recommended Focus:** Content quality at each tier

**Priority 1: Fix Synthesis Quality**

**Problem:** generatePattern() produces "important pattern for review" 56+ times.

**Solution:**
1. **Use GeneratePatternLLM by default** (already exists, not default)
2. **Validate LLM output:**
   - Is this actionable? (test: does it tell me what to DO?)
   - Is this specific? (test: could I follow this rule 6 months from now?)
   - Is this non-redundant? (test: similarity < 0.9 with existing CLAUDE.md)
3. **Reject noise:** If LLM can't produce quality principle, don't promote—stay in embeddings
4. **Measure improvement:** Before/after comparison of promoted entries quality

**Impact:** Reduces CLAUDE.md bloat, increases signal-to-noise ratio.

---

**Priority 2: Add Content Refinement Operations**

**Missing operations:**

**Rewrite (Improve Clarity):**
- **When:** Entry is vague, unclear, or overly verbose
- **LLM prompt:** "Rewrite this principle to be more specific and actionable: [entry]"
- **Validation:** User confirms improved clarity
- **Example:** "Be careful with timing" → "Use dependency injection (imptest) for time-sensitive code instead of time.Sleep() in tests"

**Merge-with-Context (Consolidate with Nuance):**
- **When:** 2-3 entries are similar (>0.85 similarity) but capture different nuances
- **Current:** Picks longer string, discards shorter
- **Improved:** LLM-driven consolidation: "Combine these principles while preserving all insights: [entry1, entry2, entry3]"
- **Validation:** User confirms no information loss
- **Example:** Merge "test behavior not presence" + "property-based testing" + "interface validation" → comprehensive testing principle

**Add-Rationale (Enrich with Why):**
- **When:** Entry is prescriptive but missing context
- **LLM prompt:** "Add rationale explaining *why* this matters: [entry]"
- **Example:** "Never amend pushed commits" → "Never amend pushed commits (Check git status for 'ahead of' first). Amending published history rewrites commits others may have pulled, causing merge conflicts and lost work."

**Extract-Examples (Separate Principle from Implementation):**
- **When:** Entry mixes abstract principle with concrete examples
- **Action:** Split into principle (CLAUDE.md) + examples (skill or documentation)
- **Example:** "TDD for all artifacts" + "tests for docs include word matching, semantic matching, structural tests" → principle stays, examples move to tdd-* skills

**Implementation:** Add refinement operations to interactive review (ISSUE-212) alongside promote/demote/merge/split.

**Impact:** Existing content improves over time, not just accumulated or deleted.

---

**Priority 3: Add Similarity Threshold Filtering (Embeddings)**

**Problem:** System always returns top N results even if all are poor matches.

**Solution:**
1. **Add query-time threshold parameter** (default 0.7-0.8 cosine similarity)
2. **Return empty set if all results below threshold** (don't force N results)
3. **Surface match confidence scores** prominently in retrieval output
4. **Log threshold misses:** Track queries where no results exceeded threshold (indicates gap in knowledge base)

**Impact:** Reduces context window pressure from low-quality retrievals, improves signal-to-noise.

---

**Priority 4: Increase Embedding Sequence Length (128 → 512)**

**Problem:** Model supports 512 tokens, we're using only 128—leaving capacity unused.

**Solution:**
1. **Update max_seq_length parameter** from 128 to 512
2. **Re-embed existing memories** (one-time migration) OR accept gradual improvement as new memories added
3. **Measure retrieval quality improvement** (before/after precision metrics)

**Impact:** Better semantic coherence for retrieval, especially for longer skill descriptions and complex memories.

**Trade-off:** Embedding generation slightly slower, but Haiku model selection mitigates cost impact.

---

**Priority 5: Clarify Model and Add Prefixes (if e5-small-v2)**

**Problem:** Code downloads all-MiniLM-L6-v2, but variable names suggest e5-small-v2.

**Solution:**
1. **Audit which model is actually used** (check downloaded model file)
2. **If e5-small-v2:**
   - Add "query:" prefix to search text
   - Add "passage:" prefix to stored memories
   - **Significant performance opportunity** (model-specific optimization)
3. **If all-MiniLM-L6-v2:**
   - Update variable names for clarity
   - Document that current approach is correct (no prefix needed)

**Impact:** If e5, this is a significant performance improvement for no additional cost. If all-MiniLM, this clarifies documentation.

---

**Priority 6: Richer Skill Descriptions (128 → 256-384 tokens)**

**Problem:** Current 128-token descriptions leave model capacity unused, reducing discovery accuracy.

**Solution:**
1. **Update skill descriptions to 256-384 tokens** with structure:
   - Core purpose (20 tokens)
   - Trigger contexts (40 tokens)
   - File types/domains (30 tokens)
   - When NOT to use (30 tokens)
   - Related concepts/keywords (50 tokens)
2. **Validate non-redundancy:** Similarity < 0.85 with other skill descriptions
3. **Measure discovery improvement:** Track skill activation accuracy (right skill for task?)

**Impact:** Better semantic matching for skill discovery, reduces activation failures.

**Trade-off:** 200 tokens × 25 skills = 5K tokens total (negligible cost for better discovery).

## Conclusion

The memory tier system has sophisticated machinery for moving content between tiers, but under-invests in content quality at each tier. Research across five dimensions (skills, CLAUDE.md, embeddings, hooks, audit) converges on the same insight:

> **Focus on content quality, not tier movement sophistication.**

**The unified optimization framework:**

1. **Reframe tiers as enforcement strength, not storage locations:**
   - Hooks (deterministic) > CLAUDE.md (judgment) > Skills (procedural) > Embeddings (episodic)

2. **Optimize each tier for its purpose:**
   - CLAUDE.md: Brevity + pointers + routing (target <100 lines)
   - Skills: Discoverable + actionable + bounded scope
   - Embeddings: Semantically rich + quality-filtered + high-confidence
   - Hooks: Deterministic quality gates

3. **Use model selection to optimize operations:**
   - Tier 3 (Embeddings): Haiku for generation/search, Sonnet for analysis
   - Tier 2 (Skills): Sonnet for writes, Haiku for reads
   - Tier 1 (CLAUDE.md): Sonnet for edits, Opus for promotion decisions

4. **Build measurement before optimization:**
   - User feedback loop (helpful/wrong/unclear)
   - Retrieval precision metrics
   - CLAUDE.md effectiveness tracking
   - **Can't optimize what we don't measure**

5. **Prioritize content quality operations over tier movement:**
   - Fix synthesis quality (eliminate "important pattern" × 56)
   - Add refinement operations (rewrite, merge-with-context, add-rationale)
   - Add threshold filtering (surface fewer, higher-quality retrievals)
   - Migrate CLAUDE.md content (reduce 259 → <100 lines)

**Next Steps:**
1. Implement user feedback loop (foundation)
2. Implement top 3 hooks (enforcement)
3. Add similarity threshold filtering (quality)
4. Fix synthesis quality (eliminate noise)
5. Validate skills tier value before scaling (measure first)

**The goal:** A memory system that reliably surfaces high-quality, actionable knowledge—not sophisticated plumbing that moves noise between tiers.
