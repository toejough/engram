# Investigation: Memory Relevance & Self-Correction

**Date:** 2026-03-20
**Triggered by:** Cycle 4/5 session observation — 2/3 of surfaced memories were irrelevant

## Observation

During a session working on engram (Go codebase), ~30 memories were surfaced across tool calls. About 10 were relevant (targ-related, engram-specific). About 20 were irrelevant (3D printing, UI design, spec ID conventions from other projects). The agent ignored all of them until the feedback mechanism was built mid-session.

## Three Problems Identified

### Problem 1: Wrong-context surfacing poisons effectiveness data

A memory that's valuable in context A (3D printing) surfaces during context B (Go development) because it shares generic keywords. The agent marks it irrelevant. After enough irrelevant marks, the memory reaches the Noise quadrant and gets proposed for deletion. When the user returns to context A, the memory is gone.

**Root cause chain:**
1. Keywords generated by the LLM during `learn` include generic terms ("build", "test", "error")
2. BM25 computes IDF over the small memory corpus (~50-200 memories), not general language
3. Generic terms may have moderate-to-high IDF within the small corpus
4. Memory surfaces in wrong context → accumulates irrelevant marks → proposed for deletion
5. Valuable context-specific memory is lost

**Current state:** `--irrelevant` feedback changes NO counter. There's no way to distinguish "surfaced in wrong context" from "surfaced correctly but advice not followed." Both are collapsed into the effectiveness score.

### Problem 2: Noise growth exceeds noise decay

`learn` creates memories with no hard cap per session. Each memory surfaces repeatedly across tool calls. Most surface irrelevantly. Even with the Noise quadrant removal proposals, removal requires manual `apply-proposal` approval — a human bottleneck that's never exercised.

**Current self-correction flow:**
- `effectiveness.FromMemories()` reads TOML counters → `review.Classify()` assigns quadrant → `maintain` generates proposal
- Noise quadrant → `actionRemove` proposal
- But proposals are NOT auto-applied — they require `engram apply-proposal --action remove`
- No human regularly runs this

**Net effect:** Creation rate > removal rate → noise grows indefinitely.

### Problem 3: Feedback doesn't improve future surfacing

When a memory is marked irrelevant, its effectiveness score drops. But the memory retains the same keywords. It will match the same BM25 queries and keep surfacing in the same wrong contexts. The feedback loop measures impact but doesn't fix the root cause.

**Missing self-correction:** There is no mechanism to use irrelevance feedback to refine the memory's keywords. The ideal loop:
1. Memory surfaces in wrong context → agent gives `--irrelevant` feedback
2. System identifies which keywords on the memory matched the query (caused the false positive)
3. System removes those generic keywords from the memory
4. Memory stops surfacing in that context, retains relevance in correct context

## Architecture Context

### BM25 Scoring
- **Prompt mode** scores against: title + content + principle + keywords + concepts
- **Tool mode** scores against: title + principle + anti_pattern + keywords
- BM25 constants: k1=1.5, b=0.75, IDF smoothing=0.5
- Relevance floors: 0.05 (proven), 0.20 (unproven prompt), 0.30 (unproven tool)

### Keyword Generation
- 100% LLM-generated during `learn` → `classify`
- Zero uniqueness filtering, zero stop-word removal
- No prompt guidance about discrimination/specificity

### Quadrant Classification
- Requires ≥5 evaluations to leave "Insufficient Data"
- 50% effectiveness threshold separates high/low follow-through
- Median surfaced count separates high/low surfacing
- 2x2 matrix: Working / Leech / Hidden Gem / Noise

### Feedback Flow (post-#341)
- `engram feedback --relevant --used` → `followed_count++` in TOML
- `engram feedback --relevant --notused` → `ignored_count++` in TOML
- `engram feedback --irrelevant` → **no counter change** (gap)
- `effectiveness.FromMemories()` reads TOML counters → feeds into quadrant classifier

### Per-Project Scoping
- **Not implemented.** All memories are in a flat namespace under `<data-dir>/memories/`.
- No project tagging, no CWD-based filtering.

## Proposed Solutions (Priority Order)

### 1. Track irrelevant_count separately (Low effort, High impact)
Add `irrelevant_count` field to memory TOML. Update `feedback --irrelevant` to increment it. This enables all downstream improvements — without knowing HOW OFTEN a memory surfaces irrelevantly, we can't make informed decisions about keyword quality.

**Issue:** #343

### 2. Improve keyword generation prompt (Low effort, High impact)
Update the classify LLM prompt to say: "Generate keywords that would NOT match unrelated domains. Prefer specific technical terms over general programming words. Avoid generic terms like 'test', 'error', 'build', 'function'."

**Issue:** #344

### 3. Reduce surfacing frequency (Low effort, Medium impact)
Drop PreToolUse/PostToolUse surfacing. Surface only at session start and user prompt. Reduces noise exposure from ~100×/session to ~10×/session.

**Issue:** #339 (already filed)

### 4. Post-generation keyword IDF filter (Medium effort, High impact)
After the LLM generates keywords, compute each keyword's document frequency across the existing memory corpus. Remove keywords that appear in >30% of memories (too common to be discriminating). This prevents generic keywords from entering the system.

**Issue:** #345

### 5. Feedback-driven keyword refinement (High effort, Very high impact)
When `--irrelevant` feedback is given, record the query context (tool name, tool input, user message). Identify which keywords on the memory matched the query. If those keywords are generic (high corpus frequency), remove them from the memory. The memory naturally stops surfacing in that context while retaining relevance where it matters.

**Issue:** #346

### 6. Auto-remove after sustained noise (Medium effort, Medium impact)
After a memory accumulates N irrelevant marks with zero relevant marks (e.g., 10 irrelevant, 0 relevant), auto-delete with backup. No manual `apply-proposal` needed. Requires irrelevant_count tracking (#343).

**Issue:** #347

## Dependencies

```
#343 (irrelevant_count) ← #346 (keyword refinement)
#343 (irrelevant_count) ← #347 (auto-remove)
#344 (prompt improvement) — independent
#339 (frequency reduction) — independent
#345 (IDF filter) — independent
```

## Key Insight

The fundamental issue is that **relevance** (was this the right context to surface?) and **effectiveness** (was the advice followed?) are collapsed into one metric. A memory with great advice but bad keywords gets penalized unfairly. Tracking irrelevant_count separately (#343) is the prerequisite for every other improvement.
