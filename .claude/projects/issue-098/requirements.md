# Requirements: Model Validation for Teammate Spawning

## REQ-001: Literal Task tool instructions in step next output

**Priority:** P0

`projctl step next` spawn actions (spawn-producer, spawn-qa) emit literal, copy-pasteable Task tool call parameters â€” not pseudocode the orchestrator LLM interprets. The output includes: `subagent_type`, `name`, `model`, and the full `prompt` text. The orchestrator executes the Task tool call exactly as specified.

**Rationale:** Removes the "LLM interprets pseudocode" failure mode that caused wrong-model spawns in ISSUE-89. Deterministic output eliminates orchestrator discretion.

**Traces to:** ISSUE-98

**Acceptance Criteria:**
- [ ] `projctl step next` spawn action output contains `subagent_type` field
- [ ] `projctl step next` spawn action output contains `name` field
- [ ] `projctl step next` spawn action output contains `model` field
- [ ] `projctl step next` spawn action output contains `prompt` field with full prompt text
- [ ] Output is valid Task tool call parameters with no interpretation required

---

## REQ-002: Model handshake in generated prompt

**Priority:** P0

The prompt text generated by `step next` for spawn actions instructs the teammate to respond with its model name as its very first message (before any other work). This is the handshake signal.

**Rationale:** Provides a verifiable signal the orchestrator can check before the teammate does any work. Catching wrong models early avoids wasted compute.

**Traces to:** ISSUE-98

**Acceptance Criteria:**
- [ ] Generated prompt contains explicit instruction to report model name first
- [ ] Teammate's first message contains only the model name string
- [ ] No other work is performed by the teammate before the handshake

---

## REQ-003: Expected model in step next output

**Priority:** P0

`step next` spawn action output includes an `expected_model` field containing the model name the orchestrator validates against the teammate's handshake response.

**Rationale:** The orchestrator needs a ground-truth value to compare the teammate's reported model against. This comes from the phase registry, not from LLM inference.

**Traces to:** ISSUE-98

**Acceptance Criteria:**
- [ ] `step next` spawn action output contains `expected_model` field
- [ ] `expected_model` value is sourced from the phase registry
- [ ] `expected_model` value matches the model specified in the spawn parameters

---

## REQ-004: Orchestrator validates model response

**Priority:** P0

After spawning a teammate, the orchestrator checks the teammate's first message against `expected_model`. On match, the spawn is successful. On mismatch, the orchestrator reports `step complete --action <action> --status failed`.

**Rationale:** Closes the loop â€” without validation, wrong-model spawns go undetected until the teammate produces incorrect artifacts.

**Traces to:** ISSUE-98, REQ-003

**Acceptance Criteria:**
- [ ] Orchestrator compares teammate's first message to `expected_model`
- [ ] Exact string match marks spawn as successful
- [ ] Mismatch triggers `step complete --action <action> --status failed`
- [ ] No teammate work proceeds until validation passes

---

## REQ-005: PairState tracks spawn_attempts

**Priority:** P1

`PairState` struct gains a `SpawnAttempts int` field (`toml:"spawn_attempts"`) that tracks how many times a spawn action has been attempted for the current sub-phase (producer or QA).

**Rationale:** The retry/escalation logic needs a counter to know when to stop retrying and escalate.

**Traces to:** ISSUE-98

**Acceptance Criteria:**
- [ ] `PairState` struct contains `SpawnAttempts int` field with `toml:"spawn_attempts"` tag
- [ ] Field persists to and loads from `state.toml`
- [ ] Field increments by 1 on each failed spawn attempt

---

## REQ-006: Retry on model mismatch (up to 3 attempts)

**Priority:** P1

When `step complete` receives a failed spawn action, it increments `spawn_attempts` in PairState. On the next `step next` call, if `spawn_attempts < 3`, the same spawn action is re-emitted (retry).

**Rationale:** Transient model-selection failures are retried before escalating to the user. Three attempts balances persistence with not spinning forever.

**Traces to:** ISSUE-98, REQ-005

**Acceptance Criteria:**
- [ ] `step complete` with failed spawn increments `spawn_attempts` by 1
- [ ] `step next` re-emits the same spawn action when `spawn_attempts < 3`
- [ ] Re-emitted spawn action is identical to the original

---

## REQ-007: Escalate to user after 3 failed spawns

**Priority:** P1

When `spawn_attempts >= 3` and the spawn fails again, `step next` emits an `escalate-user` action instead of retrying. The escalation message includes which model was expected and what models were reported.

**Rationale:** After 3 failures, the problem is systemic (e.g., model not available) and requires human intervention.

**Traces to:** ISSUE-98, REQ-006

**Acceptance Criteria:**
- [ ] `step next` emits `escalate-user` action when `spawn_attempts >= 3`
- [ ] Escalation message contains the expected model name
- [ ] Escalation message contains the model names reported by failed teammates
- [ ] No further retry is attempted after escalation

---

## REQ-008: Retry logic is orchestrator-level only

**Priority:** P1

The retry/escalation state machine (REQ-005 through REQ-007) is implemented in `step next`/`step complete` and executed by the orchestrator/team-lead. Producer and QA teammates are unaware of retry logic â€” they respond to spawn instructions without knowledge of retries.

**Rationale:** Keeps teammate skills simple and stateless. The orchestrator owns coordination; teammates own their artifact work.

**Traces to:** ISSUE-98

**Acceptance Criteria:**
- [ ] No retry logic exists in producer or QA teammate skills
- [ ] `step next` and `step complete` contain all retry/escalation logic
- [ ] Teammates receive identical prompts on retry (no retry-specific instructions)

---

## REQ-009: Spawn attempts reset on successful spawn

**Priority:** P1

When a spawn action completes with status `done` (model validated), `spawn_attempts` resets to 0 in PairState.

**Rationale:** Each sub-phase (producer, QA) gets its own fresh retry budget. A successful producer spawn does not consume the QA spawn's retry budget.

**Traces to:** ISSUE-98, REQ-005

**Acceptance Criteria:**
- [ ] `step complete` with status `done` sets `spawn_attempts` to 0
- [ ] Reset persists to `state.toml`
- [ ] Subsequent sub-phase starts with `spawn_attempts` at 0

---

## REQ-010: Backward compatibility with existing PairState

**Priority:** P2

Existing `state.toml` files without `spawn_attempts` in their PairState entries continue to work. `spawn_attempts` defaults to 0 when absent.

**Rationale:** Existing projects are not broken by this change. Go's zero-value semantics handle this naturally for int fields.

**Traces to:** ISSUE-98

**Acceptance Criteria:**
- [ ] `state.toml` files without `spawn_attempts` load without error
- [ ] Missing `spawn_attempts` defaults to 0
- [ ] No migration step is required for existing projects
