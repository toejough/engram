## Design: Dep Group Chaining

### Data Model

Replace the flat representation with ordered groups:

```go
type depGroup struct {
    targets []*Target
    mode    DepMode
}

// In Target struct:
depGroups []depGroup   // replaces deps + depMode
```

### API Behavior

`Deps()` determines mode from args (serial default). If the last group has the same mode, coalesce. Otherwise, append a new group.

```go
// Coalescing:
t.Deps(a).Deps(b)                    // 1 serial group: [a, b]
t.Deps(a, Parallel).Deps(b, Parallel) // 1 parallel group: [a, b]

// Chaining:
t.Deps(a).Deps(b, c, Parallel).Deps(d) // 3 groups: serial[a], parallel[b,c], serial[d]
```

### Execution

Iterate groups in order. Each group completes before the next starts. Within a group, serial runs sequentially, parallel runs concurrently.

### Accessor Changes

- `GetDeps()` — returns all targets flattened (backward compatible)
- `GetDepMode()` — returns mode if all groups share it, `DepModeMixed` if mixed
- **New**: `GetDepGroups()` returns `[]DepGroup` (exported view struct) for consumers that need the full picture

### Help Display

Currently: `Deps: lint, test (parallel)`

With chaining: `Deps: generate → lint, test (parallel) → deploy`

Serial groups list targets comma-separated. Parallel groups add `(parallel)`. Groups separated by `→`.

### `--dep-mode` Override

Currently parsed but not applied at runtime. With chaining, `--dep-mode serial` would flatten all groups into one serial group. `--dep-mode parallel` flattens to one parallel group. This makes the override actually work.

## What is Ralph Loop?

Ralph Loop implements the Ralph Wiggum technique - an iterative development methodology based on continuous AI loops, pioneered by Geoffrey Huntley.

**Core concept:**
```bash
while :; do
  cat PROMPT.md | claude-code --continue
done
```

The same prompt is fed to Claude repeatedly. The "self-referential" aspect comes from Claude seeing its own previous work in the files and git history, not from feeding output back as input.

**Each iteration:**
1. Claude receives the SAME prompt
2. Works on the task, modifying files
3. Tries to exit
4. Stop hook intercepts and feeds the same prompt again
5. Claude sees its previous work in the files
6. Iteratively improves until completion

The technique is described as "deterministically bad in an undeterministic world" - failures are predictable, enabling systematic improvement through prompt tuning.

## Available Commands

### /ralph-loop <PROMPT> [OPTIONS]

Start a Ralph loop in your current session.

**Usage:**
```
/ralph-loop "Refactor the cache layer" --max-iterations 20
/ralph-loop "Add tests" --completion-promise "TESTS COMPLETE"
```

**Options:**
- `--max-iterations <n>` - Max iterations before auto-stop
- `--completion-promise <text>` - Promise phrase to signal completion

**How it works:**
1. Creates `.claude/.ralph-loop.local.md` state file
2. You work on the task
3. When you try to exit, stop hook intercepts
4. Same prompt fed back
5. You see your previous work
6. Continues until promise detected or max iterations

---

### /cancel-ralph

Cancel an active Ralph loop (removes the loop state file).

**Usage:**
```
/cancel-ralph
```

**How it works:**
- Checks for active loop state file
- Removes `.claude/.ralph-loop.local.md`
- Reports cancellation with iteration count

---

## Key Concepts

### Completion Promises

To signal completion, Claude must output a `<promise>` tag:

```
<promise>TASK COMPLETE</promise>
```

The stop hook looks for this specific tag. Without it (or `--max-iterations`), Ralph runs infinitely.

### Self-Reference Mechanism

The "loop" doesn't mean Claude talks to itself. It means:
- Same prompt repeated
- Claude's work persists in files
- Each iteration sees previous attempts
- Builds incrementally toward goal

## Example

### Interactive Bug Fix

```
/ralph-loop "Fix the token refresh logic in auth.ts. Output <promise>FIXED</promise> when all tests pass." --completion-promise "FIXED" --max-iterations 10
```

You'll see Ralph:
- Attempt fixes
- Run tests
- See failures
- Iterate on solution
- In your current session

## When to Use Ralph

**Good for:**
- Well-defined tasks with clear success criteria
- Tasks requiring iteration and refinement
- Iterative development with self-correction
- Greenfield projects

**Not good for:**
- Tasks requiring human judgment or design decisions
- One-shot operations
- Tasks with unclear success criteria
- Debugging production issues (use targeted debugging instead)

- Ralph Orchestrator: https://github.com/mikeyobrien/ralph-orchestrator

[assistant] Nothing, really. It's an honor system. If the prompt doesn't have clear, verifiable success criteria, Claude can (and likely will) optimistically claim it's done early.

The mitigation is writing prompts with **objectively checkable gates** — like "all tests pass" or "linter reports zero errors" — so Claude has to actually verify before it can honestly output the promise. That's why the original author describes it as "deterministically bad in an undeterministic world" — you tune the prompt iteratively to close loopholes.

## Design: Help System Overhaul

### 1. Flag Mode Tags

Every targ flag in `internal/flags/flags.go` gets an explicit `Mode` field:

```go
type FlagMode int
const (
    FlagModeAll      FlagMode = iota // Both targ CLI and binary
    FlagModeTargOnly                 // Only targ CLI
)
```

Current flags classified:

| Mode | Flags |
|------|-------|
| `FlagModeAll` | `--help`, `--completion` |
| `FlagModeTargOnly` | Everything else (`--timeout`, `--parallel`, `--create`, `--source`, etc.) |

A test enforces every `flags.Def` has `Mode` explicitly set (zero value `FlagModeAll` is fine for help/completion, but the test checks that every flag was *consciously* assigned).

### 2. Binary Mode Propagation

`targ.Main()` sets `BinaryMode = true` on `RunOptions`. This flows through to `TargFlagFilter.BinaryMode`, which `shouldSkipTargFlag` already checks — it just never gets set today.

### 3. Help Labels

| Context | Usage line | Flag section header |
|---------|-----------|-------------------|
| Targ CLI root | `targ [global flags...] <command>...` | "Global flags:" |
| Targ CLI target | `targ [global flags...] deploy [flags...]` | "Global flags:" + "Flags:" |
| Binary root | `mybin [flags...] <command>...` | "Flags:" |
| Binary target | `mybin greet [flags...]` | "Flags:" |

### 4. Auto-Generated Examples

A new function generates examples from command metadata. It replaces the hardcoded examples. Logic:

**Root help (targ CLI):**
- "Run a command:" → `targ <first-command>`
- "Chain commands:" → `targ <cmd1> <cmd2>` (if 2+ commands exist)
- One global flag example (e.g., `targ <cmd> --timeout 30s`)

**Root help (binary):**
- "Run a command:" → `mybin <first-command>` (if subcommands exist)
- OR basic usage if no subcommands

**Target help (either mode):**
- "Basic usage:" → `<bin> <cmd>` with required positional args
- "With options:" → `<bin> <cmd>` with required args + one optional flag (if any exist)
- Omit examples that would be identical to "basic usage"

**Type-aware example values:**
- `string` → use the placeholder text (e.g., `VALUE`, `PATH`) 
- `int` → `10`
- `float64` → `0.5`
- `bool` → just the flag name (no value)
- `duration` → `30s`

**User override:** If `.Examples()` is called on a target, those replace auto-generated examples entirely.

### 5. No Default Examples in Binary Mode (Without Subcommands)

If a binary has no subcommands and no flags beyond `--help`/`--completion`, skip the examples section entirely. Nothing useful to show.
