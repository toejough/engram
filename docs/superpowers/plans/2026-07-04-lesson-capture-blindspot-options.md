# Lesson-Capture Blind-Spot — Options Plan

**Goal:** diagnose why the cycle's two self-discovered failure modes were crystallized only after Joe asked, and present guard options. Deliverable = an options doc; PRESENT AND STOP — no skill edits without Joe's pick.

**Ask (Joe, verbatim):** "I don't love that I had to point those out. Is there a category error or some blind spot we can guard against in the learn skill, or somewhere else? /please think about this and present some options"

## Diagnosis (from orientation — the doc presents this as the pipeline trace)

- **The category error is in the learn skill's Step-2 taxonomy:** it scans exactly two moment kinds — USER corrections and explicit save-requests — and its red-flags table forbids writing anything else ("things nobody asked you to remember"). Self-discovered reversals (a design superseded, a finding retro-invalidated) match neither kind: nobody SAID the correction, so the scan passes over them.
- **Aggravator 1 — "handled" bias:** both failures were corrected in repo docs mid-cycle (CORRECTION section, postscript). A corrected record feels done; but record-correction ≠ lesson-capture (note 76: you can only fix a conclusion that exists as a note; docs are chunk-memory, not recall-ranked lessons).
- **Aggravator 2 — no mid-cycle hook:** learn fires at cycle open/close; discoveries at execution time land in working docs only.
- **Aggravator 3 — ungated escalations:** the faulty 0/27 claim reached Joe via a mid-cycle AskUserQuestion; Gate D covers commit prose at step 6 but no gate reviews mid-cycle escalation payloads carrying measured claims (note 162's re-derive-numbers rule stops at gates over DOCS).
- **Recurrence evidence:** note 154 (2026-07-01) records the SAME family — Joe had to ask for a parked observation to be recorded. Today = instance two. Crystallized as note 169 (supersedes 154, updates).
- **Constraint (note 147):** Joe rejected recall-side after-failure cues 2026-06-30. All options here are CAPTURE-side (learn/please), not recall-firing changes — named explicitly in the doc.

## Options to present (full set, honest ratings; costs include note-26 writing-skills TDD and note-28 plan-doc pressure tests where they apply)

- **G1 — learn taxonomy third kind: REVERSALS.** Step-2 scan adds: "a conclusion, design, or verdict you PRESENTED (to the user, a gate, or a plan) was later overturned — by you, a reviewer, or an instrument → crystallize the root cause of the original error." Plus a red-flags row ("You corrected a repo doc and skipped the note — record-correction is not capture"). Fixes the category error directly. Implementation note (Gate A F1): Step 2's "exactly two kinds" header (SKILL.md:55) and the line-110 red-flag are themselves change targets; the alternative shape is a new numbered sub-step per the Step-2.5 precedent — the build's TDD decides which. Weakness: still relies on the closing agent's honest scan (same bias window).
- **G2 — please step-7 lessons audit over the cycle's MECHANICAL corpus.** Before the cycle report: enumerate (a) pre-registered STOPs that fired, (b) gate FAILs, (c) commits whose messages contain CORRECTION/supersede/instrument-invalid/redraw, (d) AskUserQuestion escalations — and require each to map to a vault note or an explicit one-line "no lesson because X." Observable corpus (commits/gate verdicts), so it doesn't trust in-the-moment memory. Weakness: heavier step 7; the mapping judgment is still self-performed.
- **G3 — fresh-context lessons reviewer at cycle close.** A haiku/sonnet reviewer reads the cycle's commits + report draft and REFUTES: "which failure modes occurred, and is each crystallized?" Externalizes the audit — removes self-serving bias entirely (a subagent plays Joe's role before Joe has to). Weakness: +1 reviewer per cycle (~$0.05–0.15); needs the G2 corpus definition anyway to be effective.
- **G4 — crystallize-on-discovery coupling.** Rule (in please's execution discipline): writing a CORRECTION/supersession/instrument-invalid label into any repo doc REQUIRES the paired vault write in the same action (via write-memory handoff), scoped to reversals of conclusions already presented. Catches the lesson at the moment of discovery, not cycle end. Weakness: rule-in-prose enforcement (the same class that under-fired for the atom); scope discipline needed to avoid note-spam.
- **G5 — gate mid-cycle escalations.** Any AskUserQuestion/STOP report to Joe carrying MEASURED claims gets a fast fresh-context ground-truth check first (re-derive the load-bearing number from raw artifacts — note 162's rule extended from docs to escalations). This targets the REPORTING failure specifically: it would plausibly have caught the 0/27 artifact BEFORE Joe decided on it. Weakness: adds latency to escalations; needs a proportionality rule (only measured claims, only load-bearing ones).

- **G6 — escalation provenance rule (lightweight G5).** Rule in please: any MEASURED claim in a mid-cycle escalation (AskUserQuestion, STOP report) must carry its evidence pointer (file/command) plus a one-line validity statement ("verified how?"). No reviewer round-trip, no latency; catches the 0/27 class at authoring time by forcing the "verified how?" question the author skipped. Weakness: self-enforced prose rule (under-fire family) — G5 is the enforced version. (Added per Gate-A ask-alignment: the reporting side needs more than one option.)

**Recommendation shape (doc argues, Joe decides):** G1+G2 as the core (taxonomy fix + mechanical corpus so the scan can't rely on memory), G6 for the reporting side with G5 as its enforced upgrade if G6 alone proves leaky. G3 as the stronger-but-heavier alternative to G2. G4 as a complement whose enforcement risk is real (measured under-fire family).
