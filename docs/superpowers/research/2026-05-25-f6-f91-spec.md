# F6 + F9.1 — subgraph clustering at query time spec

Date: 2026-05-25. Implementation contract for the second remaining
v2 slice, per the v2 execution roadmap.

## What ships

`engram query` gains three new computations stacked on top of the
existing direct-hit ranking: 3-hop authored-wikilink expansion,
auto-k-means clustering over the expanded subgraph, and in-degree
hub identification. The CLI surface (`engram query <string>
[--limit N]`) does not change; the YAML payload gains a `clusters`
section and richer `items.provenances` values.

Rejected alternative: ship link expansion alone first (F6 without
F9.1). Without clustering, the expanded subgraph is just more
noise — the value of expansion is realized only when clusters
extract structure from it. Ship both or ship neither.

## Pipeline

```
query string
  └─► embed → top-k cosine over all compatible sidecars → direct hits (existing)
       └─► 3-hop authored-wikilink BFS from direct hits → subgraph
            ├─► k-means (k=2..7, silhouette-selected) over subgraph embeddings → clusters
            └─► in-degree count over subgraph → top-5 hubs
                 └─► merge into payload (items + clusters + budget)
```

Each stage operates on the prior stage's output. Failures are
loud (no silent fallback — see "Failure modes" below).

## Subgraph expansion (F6)

Authored-wikilink graph only — no emergent edges (F5 dropped),
no Luhmann-adjacency expansion (the wikilink structure already
encodes adjacency by construction; see `Permanent/4j`).

BFS from direct hits, depth 3, with a visited set. Cycles handled
naturally by the visited set. Edges traversed in both directions
(outgoing `Related to:` wikilinks AND inbound backlinks from other
notes' `Related to:` blocks) — the wikilink graph is undirected
for expansion. (Hub identification later uses the same graph
directed — `in_degree` counts inbound wikilinks only.)

**Subgraph cap:** 200 notes. If BFS would exceed the cap, stop
expansion at the depth that fits. This prevents pathological cases
where dense hubs explode the subgraph. The cap is loud (budget
field `subgraph_size_capped: true` when triggered, see below).

**Sidecar requirement:** subgraph membership requires a compatible
`.vec.json` sidecar (same model_id as the active embedder). Notes
without compatible sidecars are dropped from the subgraph silently
— they couldn't have been direct hits either, so the user sees a
consistent "embedded-only" view.

## Clustering (F9.1)

Algorithm: k-means over the subgraph's embedding vectors. Loop
k=2..7, compute silhouette score for each k, pick the k with the
highest silhouette. Implementation must be pure Go (no CGO).

**Library choice:** gonum has `floats` / `mat` utilities but no
packaged k-means or silhouette primitive at time of writing. Do
not burn a research cycle hunting one — write k-means and
silhouette inline in `internal/cluster/` (~50-100 LOC each, both
algorithms are short) rather than adding a third-party
dependency. The 384-dim float32 vectors and small subgraph sizes
(≤ 200) mean performance is not load-bearing.

**Distance:** cosine distance (1 - cosine_similarity). L2 is
equivalent for normalized vectors; engram's vectors are already
normalized at embed time.

**Minimum cluster size:** 2 (k-means with singletons is noise).
Singletons are absorbed into the nearest cluster post-hoc.

**Silhouette threshold:** if max silhouette across k=2..7 is
< 0.10, return zero clusters (the subgraph has no meaningful
structure). Empty `clusters: []` in the payload, not absent
section.

**Tiny-subgraph short-circuit:** if `subgraph_size < 6`, skip
clustering entirely and return `clusters: []`. With fewer than 6
notes there isn't room for two clusters of meaningful size (≥ 2
each, plus headroom for singletons to be absorbed) — the direct
hits already cover the subgraph. Loud in `clusters_found: 0`.

**Representative selection:** for each cluster, the
representative is the member closest to the cluster centroid by
cosine distance. Tie-broken by highest direct-hit score, then by
lexicographic path.

**Determinism:** k-means initialization uses a seeded PRNG
derived from a hash of the query string. Same query against the
same vault state → same cluster IDs and same member assignments.
Adding or removing notes will reshuffle (different inputs); the
guarantee is about input → output, not about ID stability across
time. This matters for test reproducibility and for user trust
(running the same query twice on an unchanged vault shouldn't
shuffle clusters).

## Hub identification (F9.1 component)

In-degree per subgraph note = count of other subgraph notes whose
`Related to:` blocks contain a wikilink to this note. Computed
once over the subgraph after expansion completes.

Top 5 by in-degree, ties broken by direct-hit score (higher
first), then lexicographic path. If fewer than 5 notes in the
subgraph have non-zero in-degree, return fewer.

Hubs appear in `items.provenances` (with `hub` value) but do not
have a separate top-level `hubs:` section — F7's resolved
payload puts hub metadata inline on the item via `in_degree:
<n>` when `hub` is in `provenances`.

## Payload changes

Existing payload (current):

```yaml
version: 1
query: "..."
items: [{path, kind, score, provenances, content}, ...]
budget: {total_notes, with_embeddings, direct_hits_returned, limit}
```

New payload (after F6+F9.1):

```yaml
version: 1
query: "..."
items:                              # front-loaded, dedup'd
  - path: Permanent/...
    kind: fact
    score: 0.85
    provenances: [direct, cluster_rep, hub]
    cluster_id: 0                   # iff cluster_rep ∈ provenances
    in_degree: 9                    # iff hub ∈ provenances
    content: |
      <full text, wikilinks stripped>
  - ...
clusters:                           # may be empty []
  - id: 0
    size: 12
    silhouette: 0.43
    members:
      - path: Permanent/...
        score: 0.85
        is_representative: true
      - path: Permanent/...
        score: 0.71
        is_representative: false
      - ...
  - ...
budget:
  total_notes: 480
  with_embeddings: 480
  subgraph_size: 67
  subgraph_size_capped: false       # true iff BFS hit the 200 cap
  hops_traversed: 3
  clusters_found: 3
  hubs_returned: 5
  direct_hits_returned: 20
  items_with_full_content: 28
  limit: 20
```

**Item ordering** (per F7 resolution):
1. Provenance count descending (more provenances first).
2. Highest-priority provenance as tiebreak (`direct` > `cluster_rep` > `hub`).
3. Score descending as final tiebreak.

**Item dedup:** a path appears once in `items` regardless of role
count. The `provenances` list names the roles.

**Full-content scope:** every direct hit + every cluster
representative + every hub gets `content:`. Non-rep cluster
members get path + score only inside `clusters.members`.

**Budget interpretation:** `subgraph_size` is post-cap (always
≤ 200). `subgraph_size_capped: true` signals the cap fired.
`hops_traversed` is the depth BFS actually reached (≤ 3; may
be smaller if the cap stopped expansion earlier).

## CLI surface

No changes. `engram query <string> [--limit N]` keeps its current
signature. `--limit` continues to cap the **direct hits** considered
for expansion; cluster reps and hubs may appear in `items` beyond
the limit (so a `--limit 20` query can yield 28 full-content items
when there are 5 hubs and 3 cluster reps not already in the top
20 direct hits).

This matches F7's resolved spec ordering: items are bounded by
"direct hits ∪ cluster reps ∪ hubs", not by `--limit` alone.

## Integration with existing query pipeline

The current `RunQuery` flow stays the spine:

1. `loadCompatibleSidecars` (unchanged).
2. `rankCandidates` (unchanged — produces the top-k direct hits).
3. **NEW:** `expandSubgraph` — BFS from direct hits over the
   authored wikilink graph (uses `vaultgraph.Note.Wikilinks` /
   backlinks).
4. **NEW:** `clusterSubgraph` — k-means + silhouette over
   subgraph embeddings.
5. **NEW:** `identifyHubs` — in-degree count, top-5.
6. **NEW:** `mergeProvenances` — annotate items with roles,
   dedup, sort per F7.
7. `renderQueryPayload` extended with `clusters` + new budget
   fields.

No changes to `QueryDeps`. The wikilink graph traversal uses the
existing `vaultgraph` package; if missing primitives, add them
there (not in `cli/`).

## Failure modes

- **Empty subgraph** (no direct hits have wikilinks, or k=2 isn't
  reachable): payload has `clusters: []`, `hubs_returned: 0`.
  Items remain the direct hits only. Loud in the budget; not an
  error.
- **All-incompatible sidecars in subgraph** (the direct hits
  point to notes with stale sidecars): same as direct-hits guard
  — `loadCompatibleSidecars` already gates this. Subgraph stays
  empty if no expansion produces a compatible neighbor.
- **k-means non-convergence**: cap iterations at 100. If no
  convergence, return the best assignment found and log a
  warning (don't fail the query).
- **Hash collision on seeded PRNG**: not a real failure mode —
  collisions just mean two queries share initialization; output
  still valid for each.

## Integration notes

`vaultgraph` may need: a `Wikilinks(noteBasename) []string` and
`Backlinks(noteBasename) []string` API (or single
`Neighbors(noteBasename) []string` that returns both). Reuse if
present; extend if missing. Do not duplicate graph state
in `cli/`.

The k-means + silhouette code stays in `internal/embed/` or a new
`internal/cluster/` package — not in `cli/`. The `cli/query.go`
file calls into it. This keeps the algorithmic core testable
independent of the CLI plumbing.

## Out of scope

Explicit silences (per "structured artifact silences are
deliberate design"):

- Auto-synthesis at query time. Engram returns clusters; the
  consuming skill (an updated `/recall`) decides whether to
  synthesize and write via `engram learn fact|feedback`. F9.1
  *auto-synthesis* stays deferred to v3.
- Whole-vault clustering. v2 clusters only the per-query
  subgraph (~30-200 notes). Whole-vault clustering is v3.
- Block-level / field-level embedding granularity (F9.2).
  Stays deferred.
- Cross-link emergent edges (F5 dropped). The graph for
  expansion is authored wikilinks only.
- Co-retrieval counters or query logs. No new state files;
  every query computes from scratch.
- Cluster persistence. Clusters are computed per-query and
  discarded after the payload is rendered. No `clusters.json`,
  no caching.
- Cross-cluster relations. v2 returns flat clusters; cluster
  hierarchies / nested clusters stay out.
- Query-time re-ranking based on cluster membership.
  `items` are still ordered by F7's rules (provenance count
  → priority → score), not by cluster.
- Configurable hop depth or cluster k range. Hop depth fixed at
  3, k range fixed at 2..7. Future tuning is its own decision.
