# projctl Deep Review - January 2025

## Executive Summary

This document provides a comprehensive review of projctl and its skills ecosystem, comparing it with alternative agent orchestration solutions, and recommending a path forward based on your stated goals.

**Your stated goals:**

1. Dependable Claude Code agent orchestrator
2. Maximum autonomy with minimum human intervention
3. Cheapest agents + smallest context possible
4. Dedicated deterministic tooling over LLM judgment
5. Learn from corrections
6. Push for behavioral correctness (with tests)
7. Confident traceability from idea → implementation
8. Support: adopt existing codebases, align, new projects, from simple to complex

## Part 2.5: Vercel Research - Passive Context vs Skills

A [January 2025 Vercel study](https://vercel.com/blog/agents-md-outperforms-skills-in-our-agent-evals) found dramatic differences between passive context (AGENTS.md) and skill-based approaches.

### Performance Data

| Approach                      | Pass Rate | Notes                          |
| ----------------------------- | --------- | ------------------------------ |
| Baseline (no docs)            | 53%       | Pre-training only              |
| Skills (default)              | 53%       | +0pp - no improvement          |
| Skills (with explicit prompt) | 79%       | +26pp - requires careful setup |
| AGENTS.md (passive context)   | **100%**  | +47pp - always available       |

**Critical finding:** In 56% of eval cases, the skill was never invoked despite being available.

### Why Passive Context Wins

1. **No decision overhead** - Agent doesn't have to decide whether to look up docs
2. **Persistent availability** - Content is in system prompt for every turn
3. **No sequencing problems** - Skills create "read docs first vs explore code first" dilemmas

### Instruction Design That Works

Wording is fragile. Two similar approaches yielded different outcomes:

| Instruction                          | Result                                            |
| ------------------------------------ | ------------------------------------------------- |
| "Invoke skill first"                 | Anchored on docs, missed project context          |
| "Explore project first, then invoke" | Built mental models before docs, performed better |

The winning instruction: **"Prefer retrieval-led reasoning over pre-training-led reasoning"**

### Compression Works

- Initial: 40KB documentation
- Compressed: 8KB (80% reduction)
- Result: Still 100% pass rate

Technique: Pipe-delimited index pointing to retrievable files, not embedding full content.

### Implications for projctl

| Current State             | Recommended Change                               |
| ------------------------- | ------------------------------------------------ |
| Skills contain all rules  | Critical rules → CLAUDE.md (passive)             |
| Skills rely on invocation | User-triggered only, not agent-initiated         |
| Full skill content        | Compressed index + on-demand retrieval           |
| Hope agent reads skills   | Enforce via deterministic CLI, not skill prompts |

**Key insight:** Skills are unreliable for agent-initiated behavior. Use them for user-triggered workflows (`/commit`, `/project`). Put critical rules in CLAUDE.md where they're always visible.

### Action Items

1. **Migrate critical rules to CLAUDE.md** - TDD discipline, traceability format, commit conventions
2. **Skills become user-triggered only** - Remove expectation that agent will self-invoke
3. **Compress skill content** - Index + retrieval, not full embedding
4. **Deterministic enforcement** - `projctl` validates rules, not skill prompts

## Part 4: Recommended Architecture

Based on analysis, here's what projctl should become:

### Core Principles (Keep)

1. **Deterministic over LLM** - CLI tools for mechanical operations
2. **Inline traceability** - `**Traces to:**` embedded in artifacts
3. **TDD discipline** - Red→Green→Refactor with commits
4. **Evidence-based audits** - Concrete proof required
5. **State machine orchestration** - Durable progress tracking

### New Capabilities (Add)

#### 1. Complete CLI Coverage

Every skill-referenced command must exist:

```
projctl context write/read    # Skill handoff
projctl escalation list/write/resolve  # Escalation workflow
projctl integrate features    # End-of-command sequence
projctl conflict create/check/list  # Conflict management
projctl corrections log/count # Correction tracking
```

#### 2. Model Routing

Add task-complexity routing to minimize cost:

```toml
# project-config.toml
[routing]
simple = "haiku"      # Grep, read, simple edits
medium = "sonnet"     # Most development work
complex = "opus"      # Architecture, meta-audit
threshold_lines = 50  # >50 LOC change = medium→complex
```

Skills declare their default model, but orchestrator can override.

#### 3. Cost Visibility

Track and report token usage:

```
projctl usage report --session
projctl usage report --project my-project
```

JSONL log already has structure for this; just needs aggregation.

#### 4. Graceful Degradation

When a skill fails:

1. Log failure with context
2. Offer retry/skip/escalate
3. Continue with unblocked tasks (already designed, needs enforcement)

#### 5. Learning Loop

Automatic skill improvement from corrections. **Critically, the orchestrator drives this—not the agent.**

```
# Orchestrator (step 0 in control loop) detects correction signals in user messages:
#   "that's wrong", "no, do X", "I said X not Y", "remember that..."
# Orchestrator calls:
projctl corrections log --message "..."  # Track correction (NOT agent)

# After skill dispatch (step 9.5 in control loop), orchestrator checks:
projctl corrections count --since=session-start

# If count >= 2, orchestrator injects /meta-audit before continuing
```

See **Phase 4 Integration Specification** for implementation details.

#### 6. Background Territory Mapping

Before main work begins, dispatch a fast/cheap agent to explore and summarize:

```
# Orchestrator dispatches haiku agent to map territory
projctl map --dir . --output context/territory.toml

# Territory map contains:
# - File structure summary
# - Key entry points identified
# - Test file locations
# - Config files found
# - Relevant patterns detected
```

**How it works:**

1. User invokes `/project new` or starts a task
2. Before main skill dispatch, orchestrator runs `Explore` agent (haiku) in background
3. Explore agent produces compressed territory map
4. Main skill receives territory map in context, skips redundant exploration
5. Result: Smaller context, faster execution, lower cost

**Where it applies:**

| Workflow         | Territory Mapping                          |
| ---------------- | ------------------------------------------ |
| `/project new`   | Map codebase before pm-interview           |
| `/project adopt` | Map existing docs + code structure         |
| `tdd-red`        | Map test patterns before writing new tests |
| `tdd-green`      | Map implementation patterns before coding  |

**Key insight from oh-my-opencode:** Their Sisyphus agent "fires off background tasks to faster, cheaper models in parallel" to keep the main agent's context lean. We adopt this pattern.

**Implementation:**

```toml
# context/territory.toml (output of mapping)
[structure]
root = "/Users/joe/repos/personal/projctl"
languages = ["go"]
build_tool = "mage"
test_framework = "go test + gomega + rapid"

[entry_points]
cli = "cmd/projctl/main.go"
public_api = "projctl.go"

[packages]
count = 12
internal = ["config", "context", "conflict", "coverage", "escalation", "integrate", "log", "parser", "screenshot", "state", "trace"]

[tests]
pattern = "*_test.go"
count = 27

[docs]
readme = "README.md"
artifacts = ["docs/requirements.md", "docs/design.md", "docs/architecture.md"]
```

This compressed map (~500 tokens) replaces extensive file exploration (~5000+ tokens).

#### 7. Parallel Skill Dispatch

Enable concurrent execution of independent skills:

```toml
# In orchestrator logic, not config
# When tasks have no dependencies, dispatch in parallel

# Example: During /project adopt, these can run concurrently:
# - pm-infer (analyzing README, tests)
# - architect-infer (analyzing package structure)
# - design-infer (analyzing CLI help)
```

**Where parallelism helps:**

| Workflow Phase  | Parallel Opportunity                                   |
| --------------- | ------------------------------------------------------ |
| adopt-infer-\*  | pm-infer, design-infer, architect-infer simultaneously |
| implementation  | Independent TASK-NNN TDD cycles                        |
| audit           | pm-audit, architect-audit in parallel                  |
| alignment-check | Scan different artifact types concurrently             |

**Implementation approach:**

The Task tool already supports parallel dispatch via multiple tool calls in one message. The orchestrator should:

1. Analyze task dependencies (already in tasks.md DAG)
2. Identify independent tasks/skills
3. Dispatch via parallel Task tool invocations
4. Collect results before proceeding to dependent work

**Not the same as swarms:** This is simple fork-join parallelism for independent work, not persistent multi-agent coordination with roles and consensus.

#### 8. Graceful Degradation

When a skill fails, the system should recover gracefully:

```
# Current: Skill fails → orchestrator confused → user intervenes
# Target: Skill fails → logged → retry/skip/escalate decision → continue

projctl state get --dir .
# Output includes: last_error, retry_count, available_actions
```

**Failure modes to handle:**

| Failure                       | Response                                 |
| ----------------------------- | ---------------------------------------- |
| Skill times out               | Log, offer retry with longer timeout     |
| Skill produces invalid output | Log, parse what's valid, escalate gaps   |
| Skill returns error status    | Log, check if retryable, escalate if not |
| Context file missing          | Regenerate from state, retry             |
| Dependency not met            | Block task, continue with unblocked work |

**State machine support:**

```toml
# state.toml additions
[error]
last_phase = "tdd-red"
last_task = "TASK-003"
error_type = "timeout"
retry_count = 1
max_retries = 3

[recovery]
available_actions = ["retry", "skip", "escalate"]
blocked_tasks = ["TASK-004", "TASK-005"]  # depend on TASK-003
unblocked_tasks = ["TASK-006"]  # can continue
```

**Implementation:** Orchestrator wraps skill dispatch in error handling, updates state with failure info, presents options to user or auto-retries based on config.

#### 9. LSP Integration for Deterministic Refactoring

Use Language Server Protocol for mechanical code operations instead of LLM judgment:

```
# Instead of: "rename this function" via LLM edit
# Use: LSP rename symbol (deterministic, safe)

projctl refactor rename --symbol "OldName" --to "NewName" --dir .
projctl refactor extract-function --file foo.go --lines 10-25 --name "helperFunc"
projctl refactor inline --symbol "unusedVar" --dir .
```

**Why LSP over LLM:**

| Operation        | LLM                   | LSP                    |
| ---------------- | --------------------- | ---------------------- |
| Rename symbol    | Might miss references | Finds all references   |
| Extract function | Might break scope     | Understands scope      |
| Find usages      | Pattern matching      | Semantic understanding |
| Safe delete      | Might miss callers    | Knows all callers      |

**Implementation approach:**

1. Shell out to `gopls` (Go), `typescript-language-server` (TS), etc.
2. Wrap common refactoring operations in `projctl refactor` commands
3. Skills call `projctl refactor` instead of making LLM edits for mechanical changes

**Scope:** Start with Go (gopls), expand to other languages as needed.

#### 10. CLAUDE.md Migration (Passive Context)

Move critical rules from skills to CLAUDE.md where they're always visible:

**Current state:** Rules scattered across 23 SKILL.md files. Agent must invoke skill to see rules.

**Target state:** Critical rules in CLAUDE.md (always visible), skills contain workflow logic only.

**Migration candidates:**

| Rule                                   | Current Location              | Move to CLAUDE.md?            |
| -------------------------------------- | ----------------------------- | ----------------------------- |
| TDD discipline (no test weakening)     | tdd-green, task-audit         | ✅ Yes - critical             |
| Traceability format (`**Traces to:**`) | alignment-check, trace skills | ✅ Yes - critical             |
| Commit trailer format                  | commit                        | ✅ Yes - critical             |
| Evidence-based findings only           | all audit skills              | ✅ Yes - critical             |
| No blanket linter overrides            | tdd-refactor                  | ✅ Yes - already in CLAUDE.md |
| Interview phase structure              | pm/design/architect-interview | ❌ No - workflow, not rule    |
| DAG dependency format                  | task-breakdown                | ❌ No - workflow, not rule    |

**Compression strategy (from Vercel):**

Instead of duplicating full rules, CLAUDE.md can reference skill files:

```markdown
## Critical Rules (see skills for full details)

- **TDD discipline**: Never weaken tests to pass. See `~/.claude/skills/tdd-green/SKILL.md`
- **Traceability**: Use `**Traces to:**` inline, never traceability.toml. See `~/.claude/skills/alignment-check/SKILL.md`
```

This keeps CLAUDE.md small while ensuring critical rules are always visible.

#### 11. Skill Compression (Index + Retrieval)

Reduce skill token usage via compressed indices:

**Current:** Full SKILL.md content loaded when skill invoked (~2000-5000 tokens each).

**Target:** Compressed index in SKILL.md, full details retrieved on demand.

**Example transformation:**

Before (full content):

```markdown
# TDD Green Skill

## Rules

1. Write minimal code to make tests pass
2. Do not add features beyond what tests require
3. Do not refactor yet - that's the next phase
   ... (500 more lines)
```

After (compressed index):

```markdown
# TDD Green Skill

## Quick Reference

- Minimal code only | No extra features | No refactoring yet
- On failure: check test assertions → implementation logic → dependencies
- Output: all tests pass, no new tests added

## Full Documentation

For detailed rules, examples, and edge cases: `projctl skill docs tdd-green`
```

**Implementation:**

1. Each skill has compressed SKILL.md (~500 tokens) + full SKILL-full.md
2. `projctl skill docs <name>` outputs full documentation
3. Skills can request full docs via CLI when needed
4. Orchestrator injects compressed version by default

#### 12. Relentless Continuation ("Won't Quit")

The orchestrator should continue working until blocked or complete, not stop after each phase.

**Current problem (from session logs):**

```
# Actual log pattern observed:
Done. Created 2 commits:
1. b0abe2e `refactor: move impgen/run to internal/run`
2. 3621051 `refactor(match): move implementation to internal/core`

Branch is ahead of origin by 2 commits.

No response requested.
# ← STOPS HERE, user must prompt to continue
```

**Target behavior:**

```
# Sisyphus-style continuation:
Done. Created 2 commits for TASK-003.

Checking next task... TASK-004 is unblocked.
Starting TASK-004: Add validation logic.
# ← CONTINUES AUTOMATICALLY
```

**When to stop (legitimate blockers):**

| Blocker Type                            | Action                        |
| --------------------------------------- | ----------------------------- |
| All tasks complete                      | Stop, report summary          |
| Escalation needed                       | Stop, present question        |
| Validation failed after 3 retries       | Stop, present options         |
| Ambiguous requirement                   | Stop, ask clarifying question |
| External dependency (user input needed) | Stop, explain what's needed   |

**When NOT to stop:**

| Situation                        | Current | Target                     |
| -------------------------------- | ------- | -------------------------- |
| Task complete, more tasks exist  | Stops   | Continue to next           |
| Phase complete, next phase ready | Stops   | Continue to next phase     |
| Tests pass                       | Stops   | Move to refactor phase     |
| Commit done                      | Stops   | Continue to next TDD phase |

**Implementation:**

```markdown
# In /project SKILL.md, add explicit continuation rule:

## Continuation Rule (CRITICAL)

After completing ANY task or phase:

1. Check `projctl state get` for next action
2. If unblocked work exists → continue immediately
3. If all work blocked → present blockers to user
4. If all work complete → run end-of-command sequence, report summary

NEVER say "No response requested" or stop with just a summary.
NEVER ask "Should I continue?" if next steps are clear.
ALWAYS continue until hitting a legitimate blocker.
```

**Deterministic enforcement:**

```bash
# projctl could enforce this by returning next action:
projctl state next --dir .

# Output:
# {
#   "action": "continue",
#   "next_task": "TASK-004",
#   "next_phase": "tdd-red",
#   "reason": "TASK-003 complete, TASK-004 unblocked"
# }

# Or:
# {
#   "action": "stop",
#   "reason": "escalation_pending",
#   "escalation": "ESC-001: Clarify authentication requirement"
# }
```

The orchestrator should call `projctl state next` after each completion and follow its directive.

#### 13. Cross-Project Memory System

Persistent memory for learnings, patterns, and context across projects and sessions.

**Structure:**

```
~/.claude/memory/
├── index.md           # Structured learnings (always greppable)
├── sessions/          # Compressed session summaries
│   ├── 2025-01-31-projctl-review.md
│   └── 2025-01-30-imptest-refactor.md
├── decisions/         # Decision logs (richer than events, leaner than streaming)
│   ├── 2025-01-31-projctl.jsonl
│   └── 2025-01-30-imptest.jsonl
└── embeddings.db      # SQLite-vec for semantic search
```

**Three-tier capture:**

| Tier      | Captured             | Example                                                                                                              |
| --------- | -------------------- | -------------------------------------------------------------------------------------------------------------------- |
| Events    | State transitions    | `{"subject":"task-status","message":"TASK-003 complete"}`                                                            |
| Decisions | Choices + reasoning  | `{"type":"decision","choice":"extract methods","reason":"reduces coupling","alternatives":["inline","new package"]}` |
| Summary   | Compressed narrative | "Refactored parser using method extraction. Considered new package but coupling was acceptable."                     |

**Memory types:**

| Type               | Storage        | Retrieval                  |
| ------------------ | -------------- | -------------------------- |
| Explicit learnings | index.md       | grep, always available     |
| Session context    | sessions/\*.md | grep by date/project       |
| Semantic queries   | embeddings.db  | "What do we know about X?" |

**CLI interface:**

```bash
# WRITE (called by orchestrator, not agent - see Part 4.5)
# Extract decisions from skill result file:
projctl memory extract --result context/TASK-001-tdd-green.result.toml

# Manual learning (for explicit "remember this" from user)
projctl memory learn --message "Event sourcing was overkill for project-A scale"

# Session summary (auto-generated by orchestrator at session end)
projctl memory session-end --project projctl

# READ (called by orchestrator before skill dispatch)
# Query memory semantically
projctl memory query "authentication patterns we've used"

# Query memory structurally
projctl memory grep "project-A"

# Query decisions specifically
projctl memory decisions --context "refactor"
```

**Key design:** Writing is done by the ORCHESTRATOR extracting from structured results. The agent doesn't call write commands directly (unreliable per Vercel findings). Reading is done by the orchestrator before skill dispatch to inject relevant context.

**Required skill result format:**

```toml
# Skills MUST include decisions/learnings in result.toml
[decisions]
[[decisions.items]]
context = "parser implementation"
choice = "recursive descent"
reason = "simpler than table-driven"
alternatives = ["table-driven", "parser combinator"]

[learnings]
[[learnings.items]]
content = "Property tests caught edge case with empty input"
```

The orchestrator extracts these automatically after each skill completes.

**Integration points:**

| Workflow               | Memory Query                                |
| ---------------------- | ------------------------------------------- |
| `/project new`         | "Similar projects we've built?"             |
| `/project adopt`       | "Patterns from similar codebases?"          |
| `/architect-interview` | "What worked/didn't in past architectures?" |
| Any blocker            | "Have we solved this before?"               |

**Embedding approach:**

- Use SQLite-vec (no server, single file)
- Local embeddings via ONNX (no API calls)
- Embed session summaries + explicit learnings
- Query returns relevant memories for context injection

#### 14. Deterministic Workflow Enforcement

Ensure critical workflow steps run via CLI guards, not agent judgment:

**Problem:** Orchestrator says "dispatch alignment-check after artifact phase" but agent might skip it.

**Solution:** State transitions require preconditions:

```bash
# This should FAIL if alignment-check hasn't run
projctl state transition --dir . --to architect-complete

# Error: Cannot transition to architect-complete
# Required: alignment-check must pass
# Run: projctl trace validate --dir .
```

**Enforcement rules:**

| Transition                                | Precondition                            |
| ----------------------------------------- | --------------------------------------- |
| `pm-complete` → `design-interview`        | requirements.md exists with REQ-NNN IDs |
| `design-complete` → `architect-interview` | design.md exists with DES-NNN IDs       |
| `architect-complete` → `task-breakdown`   | `projctl trace validate` passes         |
| `tdd-red` → `tdd-green`                   | Test files exist and fail               |
| `tdd-green` → `tdd-refactor`              | All tests pass                          |
| `task-complete` → next task               | `projctl trace validate` passes         |

**Implementation:**

```go
// In state/transitions.go
func (s *State) CanTransition(to Phase) error {
    switch to {
    case PhaseArchitectComplete:
        if err := validateTraceability(s.Dir); err != nil {
            return fmt.Errorf("cannot transition: %w", err)
        }
    case PhaseTDDGreen:
        if err := validateTestsFail(s.Dir, s.CurrentTask); err != nil {
            return fmt.Errorf("cannot transition: %w", err)
        }
    }
    return nil
}
```

**Key insight:** The agent can't skip what the CLI won't allow. Deterministic guards > hoping agent follows instructions.

## Part 4.5: Architecture Coherence Analysis

This section identifies integration gaps and clarifies ownership between deterministic (projctl) and agentic (LLM) components.

### Gap 1: Features Built But Not Consumed

Several features are defined but lack clear integration points:

| Feature              | Built                        | Consumed By | Gap                                |
| -------------------- | ---------------------------- | ----------- | ---------------------------------- |
| Territory mapping    | `projctl map`                | ?           | Who injects it into context? When? |
| Memory query         | `projctl memory query`       | ?           | Who calls it? When? Automatically? |
| Session summaries    | `projctl memory session-end` | ?           | What triggers this?                |
| Decision logging     | `projctl memory decide`      | ?           | Agent must call it - will it?      |
| Model routing config | `project-config.toml`        | ?           | Who reads and applies it?          |
| Cost tracking        | `projctl usage report`       | ?           | Informational only, no action      |

**Root problem:** These features require the agent to call them, but per Vercel findings, agents only invoke skills 44% of the time. If the agent doesn't call `projctl memory decide`, no decisions get logged.

### Gap 2: Missing Orchestration Control Loop

The document describes WHAT features exist but not HOW they fit together. Missing:

**What happens at each state transition?**

```
# Undefined: The actual control loop
1. Orchestrator is at phase X
2. What does it DO?
3. What checks does it run?
4. What gets injected into context?
5. What happens after skill returns?
```

**Proposed control loop (deterministic):**

```
LOOP:
  0. CLASSIFY USER MESSAGE → if correction signal, log via projctl corrections log
  1. projctl state get → current phase, task
  2. projctl state preconditions → check if transition allowed
  3. projctl map → generate territory (if not cached)
  4. projctl memory query → retrieve relevant memories
  5. projctl context write → assemble context with territory + memories
  6. DISPATCH SKILL (agentic)
  7. projctl context read --result → get skill output
  8. projctl memory decide/learn → log decisions (FROM RESULT, not agent call)
  9. projctl state next → determine continue/stop
  9.5. projctl corrections count → if >= threshold, inject /meta-audit
  10. IF continue: GOTO 1
  11. IF stop: present reason to user
```

**Key insight:** Steps 0-5 and 7-11 are DETERMINISTIC (projctl does them). Only step 6 is agentic.

### Gap 3: Ownership Ambiguity

| Feature                     | Currently        | Should Be                   | Reason                   |
| --------------------------- | ---------------- | --------------------------- | ------------------------ |
| Call `projctl map`          | Agent (implicit) | **Orchestrator (explicit)** | Agent might skip         |
| Call `projctl memory query` | Agent (implicit) | **Orchestrator (explicit)** | Agent might skip         |
| Log decisions               | Agent calls CLI  | **Parse from result.toml**  | Agent might skip         |
| Continue to next task       | Agent judgment   | **projctl state next**      | Agent stops prematurely  |
| Model selection             | Skill default    | **Orchestrator injects**    | Agent can't change model |

**Principle:** If it's critical, make it deterministic. Don't rely on agent calling things.

### Gap 4: Memory System Integration

Memory is defined but not wired in:

**Current (undefined):**

```
Memory exists → ??? → Agent somehow uses it
```

**Proposed integration:**

```
# In orchestrator control loop (deterministic):

before_skill_dispatch(phase, task):
    territory = projctl map --dir . --cached

    # Query memory based on phase
    if phase == "architect-interview":
        memories = projctl memory query "architecture decisions"
    elif phase == "tdd-red":
        memories = projctl memory query "test patterns for {task.description}"
    else:
        memories = projctl memory query "{task.description}"

    # Inject into context
    context.territory = territory
    context.relevant_memories = memories

    projctl context write --with-territory --with-memories

after_skill_result(result):
    # Extract decisions from structured result (not agent CLI calls)
    for decision in result.decisions:
        projctl memory decide --from-json decision

    for learning in result.learnings:
        projctl memory learn --from-json learning
```

**Key change:** Memory logging is extracted FROM skill results, not expecting agent to call CLI during execution.

### Gap 5: Model Routing Enforcement

Model routing is defined but unenforceable:

**Current (aspirational):**

```toml
[routing]
simple = "haiku"
medium = "sonnet"
complex = "opus"
```

**Problem:** Claude Code's model is set at session start. The orchestrator can't force mid-session model changes.

**Realistic options:**

| Option            | How                                       | Tradeoff                 |
| ----------------- | ----------------------------------------- | ------------------------ |
| Skill-level hints | Context includes `suggested_model: haiku` | Agent might ignore       |
| Subagent dispatch | Use Task tool with `model: haiku`         | Different context, works |
| Accept limitation | Log intended model, report variance       | Visibility only          |

**Recommendation:** For skills dispatched via Task tool (subagents), model routing works. For inline work, it's advisory only. Document this limitation.

### Gap 6: Cost Visibility Is Passive

Cost tracking captures data but doesn't ACT on it:

**Current:**

```
projctl usage report → shows tokens used
```

**Missing:**

- Alert when session exceeds budget
- Recommend switching to cheaper model
- Show cost impact of routing decisions

**Proposed integration:**

```bash
# In orchestrator, after each skill:
USAGE=$(projctl usage check --session)
if [ "$USAGE" -gt "$BUDGET_WARNING" ]; then
    projctl log write --level status --subject cost-warning \
        --message "Session at $USAGE tokens, consider using haiku for remaining tasks"
fi
```

### Gap 7: Decision Logging Relies on Agent

The `projctl memory decide` command expects the agent to call it. Per Vercel, this won't happen reliably.

**Solution:** Extract decisions from structured skill results.

**Skill result format (proposed):**

```toml
# context/TASK-001-tdd-green.result.toml
[status]
success = true

[outputs]
files_modified = ["parser.go", "parser_test.go"]

[decisions]

context = "parser implementation"

reason = "simpler than table-driven for our grammar"
alternatives = ["table-driven", "parser combinator"]

[learnings]
[[learnings.items]]
content = "Property tests caught edge case with empty input"
```

Orchestrator extracts `decisions` and `learnings` sections and logs them automatically.

#### Phase 4 Integration Specification (CRITICAL)

Per Vercel findings, agents won't reliably call tools. The Learning Loop must be **deterministic**, not rely on agent judgment.

**WHO logs corrections?**

The **orchestrator** detects corrections from user messages. The agent doesn't call `projctl corrections log` - the orchestrator does.

```
USER MESSAGE CLASSIFICATION:
1. Orchestrator receives user message
2. Before passing to skill, classify message type:
   - Regular input → pass through
   - Correction signal → extract correction, log, then pass through

CORRECTION SIGNALS (pattern match):
- "that's wrong"
- "no, do X instead"
- "I said X, not Y"
- "remember that..." (explicit instruction to update behavior)
- User manually calls `/audit` or `/meta-audit`

EXTRACTION:
projctl corrections log \
  --dir . \
  --message "[extracted correction content]" \
  --context "[current phase/task]" \
  --session "[session-id]"
```

This is NOT agentic - it's pattern matching on user messages before skill dispatch.

**HOW is meta-audit triggered?**

Add to the **control loop** (Part 4.5) after step 9:

```
MODIFIED CONTROL LOOP:
  1-9. [existing steps]
  9.5. projctl corrections count --since=session-start
       IF count >= threshold (default: 2):
         Inject /meta-audit as next skill before continuing
         Reset session correction counter
  10-11. [existing steps]
```

The check is **deterministic** - the orchestrator runs it, not the agent.

**Implementation details:**

```go
// In orchestrator control loop (not a skill)
func checkCorrectionThreshold(ctx Context) (bool, error) {
    count, err := exec.Command("projctl", "corrections", "count",
        "--dir", ctx.Dir,
        "--since", ctx.SessionStart,
    ).Output()

    threshold := ctx.Config.CorrectionThreshold // default: 2
    return atoi(count) >= threshold, nil
}

// After state next, before loop
if shouldTrigger, _ := checkCorrectionThreshold(ctx); shouldTrigger {
    // Inject meta-audit before continuing normal flow
    ctx.InjectSkill("/meta-audit")
    resetSessionCounter(ctx)
}
```

**What meta-audit does:**

1. Reads corrections.jsonl for patterns
2. Proposes CLAUDE.md additions
3. Returns structured result with proposed changes
4. **Orchestrator** applies changes (not agent) after user approval

**Why this works:**

| Step | Who does it | Why not agent |
|------|-------------|---------------|
| Detect correction | Orchestrator | Pattern match, not judgment |
| Log correction | Orchestrator | Deterministic after detection |
| Count check | Orchestrator | In control loop, guaranteed to run |
| Trigger meta-audit | Orchestrator | Inject into flow, not ask agent |
| Apply CLAUDE.md | Orchestrator | After user approval, deterministic |

The agent's only role is generating the CLAUDE.md proposals - the actual learning mechanics are deterministic.

## Part 6: What NOT to Build

Based on competitive analysis, these are **not** worth the complexity:

| Feature                       | Why Not                                                                       |
| ----------------------------- | ----------------------------------------------------------------------------- |
| Persistent multi-agent swarms | Complex coordination (Mayor/Polecats roles); fork-join parallelism is simpler |
| ML-driven routing             | Heuristic rules work fine for your scale                                      |
| Byzantine consensus           | You're one user, not a distributed system                                     |
| Custom vector DB              | See analysis below                                                            |
| Real-time monitoring          | JSONL logs + `tail -f` suffice                                                |

### Vector DB Analysis

**What alternatives use vector DBs for:**

| System         | Vector DB Use         | Purpose                                                  |
| -------------- | --------------------- | -------------------------------------------------------- |
| claude-flow    | RuVector (PostgreSQL) | RAG over docs, semantic code search, conversation memory |
| oh-my-opencode | Embeddings via API    | Finding similar code patterns, context retrieval         |
| Gastown        | None (git-native)     | N/A                                                      |

**Typical vector DB use cases:**

1. **Semantic search** - "Find code that handles authentication" (not just `grep "auth"`)
2. **Similar code detection** - Find functions structurally similar to this one
3. **Context retrieval** - Given current task, retrieve most relevant docs
4. **Conversation memory** - Remember past interactions across sessions

**Why we don't need it:**

| Use Case              | Vector DB Approach             | projctl Alternative                                |
| --------------------- | ------------------------------ | -------------------------------------------------- |
| Find relevant code    | Embed code → semantic search   | `grep` + Explore agent (cheaper, often sufficient) |
| Find relevant docs    | Embed docs → similarity search | Territory map + skill indices (deterministic)      |
| Remember past context | Embed conversations → retrieve | JSONL logs + corrections tracking (structured)     |
| Find similar patterns | Embed patterns → k-NN search   | LSP + AST (deterministic, exact)                   |

**Key insight:** Most "semantic search" needs in coding are actually **structural**:

- "Find all callers of this function" → LSP (exact)
- "Find files that handle X" → grep pattern + Explore agent (good enough)
- "Find similar implementations" → AST pattern matching (deterministic)

Vector DBs shine when you need fuzzy matching over natural language. Code has structure we can exploit instead.

**When we might reconsider:**

- If working across 100+ repos simultaneously
- If needing cross-project pattern learning
- If natural language documentation search becomes critical

**Current recommendation for code/docs:** Skip vector DB. Use:

1. **grep + Explore agent** for code search
2. **Territory mapping** for structural understanding
3. **CLAUDE.md + skill indices** for doc retrieval
4. **LSP** for semantic code operations

These are simpler, cheaper, and deterministic where possible.

### Memory System Gap (Reconsider Vector DB)

**The problem:** Past context and cross-project memory isn't handled well.

| Memory Type                  | Current State      | Gap                               |
| ---------------------------- | ------------------ | --------------------------------- |
| Corrections                  | JSONL logs         | Project-specific only             |
| What we did                  | Summaries in logs  | Not rich enough                   |
| Cross-project patterns       | Nothing            | Complete gap                      |
| Long-term preferences        | CLAUDE.md (manual) | Requires explicit "remember this" |
| "We tried X, it didn't work" | Lost               | Complete gap                      |

**Examples of valuable memory:**

```
# Things worth remembering that aren't corrections:

"Last month in project-A we tried event sourcing but switched to
simple state because the complexity wasn't justified for our scale."

"Joe prefers property-based tests over example-based when possible."

"The authentication refactor in project-B took 3 attempts because
we kept missing the session timeout edge case."

"When working with this codebase, always check for the legacy
API handlers in /v1/ - they have different error conventions."
```

**Why JSONL logs aren't enough:**

1. **Project-scoped** - Can't query "what did we learn across all projects?"
2. **Summary-only** - Rich context lost; just "TASK-003 complete"
3. **No semantic search** - Can't ask "what do we know about auth patterns?"
4. **Ephemeral** - Not designed for long-term retention

### Streaming Updates vs Structured Events

**The logging spectrum:**

| Approach           | What's Captured                                   | Size                  | Example                              |
| ------------------ | ------------------------------------------------- | --------------------- | ------------------------------------ |
| Full streaming     | Every message, every tool call, full conversation | Huge (~MB/session)    | Claude's ~/.claude/projects/\*.jsonl |
| Structured events  | Key transitions only                              | Small (~KB/session)   | projctl's updates.jsonl              |
| Semantic summaries | AI-compressed session narrative                   | Medium (~2KB/session) | Proposed sessions/\*.md              |

**Current projctl logging (structured events):**

```jsonl
{"timestamp":"2025-01-31T10:00:00Z","level":"status","subject":"task-status","task":"TASK-003","message":"complete"}
{"timestamp":"2025-01-31T10:05:00Z","level":"phase","subject":"phase-change","phase":"tdd-green","message":"Starting green phase"}
```

**What's lost:**

- Why TASK-003 took 20 minutes
- What approaches were tried and rejected
- The actual code reasoning
- "We considered X but chose Y because Z"

**Claude's full streaming (in project JSONL):**

```jsonl
{"type":"user","content":"Can you refactor the parser?"}
{"type":"assistant","content":"I'll analyze the parser structure first..."}
{"type":"tool_use","name":"Read","input":{"file":"parser.go"}}
{"type":"tool_result","content":"...800 lines of code..."}
{"type":"assistant","content":"I see three options: 1) Extract methods..."}
```

**What's captured:** Everything - full reasoning chain, tool calls, decisions.

**What's wrong:**

- Massive (multi-MB per session)
- No semantic structure (just raw transcript)
- Project-specific paths (not cross-project)
- Hard to search meaningfully

**The gap:**

| Need                          | Full Streaming          | Structured Events  | Neither Solves               |
| ----------------------------- | ----------------------- | ------------------ | ---------------------------- |
| "What happened in this task?" | ✅ Too much detail      | ❌ Just "complete" | -                            |
| "Why did we choose X over Y?" | ✅ Buried in transcript | ❌ Not captured    | -                            |
| "What patterns emerged?"      | ❌ Too noisy            | ❌ Not captured    | ✅ Need semantic compression |
| Cross-project search          | ❌ Project-scoped       | ❌ Project-scoped  | ✅ Need global index         |

**Proposed solution: Semantic streaming**

Capture richer updates during the session, then compress at session end:

```jsonl
// During session (richer than current, leaner than full streaming)
{"timestamp":"...","type":"decision","context":"parser refactor","choice":"extract methods","reason":"reduces coupling, easier to test","alternatives":["inline everything","new package"]}
{"timestamp":"...","type":"attempt","task":"TASK-003","approach":"recursive descent","outcome":"failed","lesson":"too complex for our grammar"}
{"timestamp":"...","type":"learning","content":"Property tests caught edge case human-written tests missed"}
```

Then at session end, compress to summary + embed for semantic search.

**Three-tier capture:**

| Tier      | When        | What                        | Storage               |
| --------- | ----------- | --------------------------- | --------------------- |
| Events    | Real-time   | State transitions (current) | updates.jsonl         |
| Decisions | During work | Choices + reasoning         | decisions.jsonl (new) |
| Summary   | Session end | Compressed narrative        | sessions/\*.md        |

This gives us:

- Fast event queries (grep updates.jsonl)
- Reasoning trail (grep decisions.jsonl)
- Semantic search (embed summaries)
- Cross-project memory (global ~/.claude/memory/)

**Options for memory system:**

| Option                                         | Pros                                         | Cons                             |
| ---------------------------------------------- | -------------------------------------------- | -------------------------------- |
| **Lightweight vector DB** (SQLite-vec, Chroma) | Semantic search, cross-project, fuzzy recall | New dependency, embedding costs  |
| **Structured memory.md**                       | Simple, readable, no deps                    | Manual curation, no fuzzy search |
| **Session summaries → searchable index**       | Compressed, greppable                        | Still no semantic matching       |
| **Hybrid: structured + vector**                | Best of both                                 | More complexity                  |

**Recommended: Hybrid approach**

```
~/.claude/
├── CLAUDE.md           # Critical rules (passive context, always loaded)
├── memory/
│   ├── index.md        # Structured: explicit learnings, preferences
│   ├── sessions/       # Compressed session summaries (greppable)
│   └── embeddings.db   # SQLite-vec for semantic search (optional)
```

**How it would work:**

1. **End of each session:** Generate summary, store in sessions/
2. **Explicit learnings:** Append to index.md (structured, always available)
3. **Semantic queries:** "What do we know about X?" → search embeddings.db
4. **Cross-project:** Memory is global (~/.claude/memory/), not per-project

**When to query memory:**

- `/project new` - "Have we built something similar before?"
- `/project adopt` - "What patterns did we learn from similar codebases?"
- Any architecture decision - "What worked/didn't work in past projects?"

**Implementation note:** SQLite-vec is lightweight (no server, single file) and can use local embeddings via ONNX. This avoids the "enterprise vector DB" complexity while solving the memory problem.

**Clarification: Parallelism vs Swarms**

| Approach              | Description                                              | Worth It? |
| --------------------- | -------------------------------------------------------- | --------- |
| Fork-join parallelism | Dispatch N independent skills, wait for all, continue    | ✅ Yes    |
| Persistent swarms     | 20-30 agents with roles, mailboxes, continuous operation | ❌ No     |

Fork-join is simple: identify independent work, dispatch in parallel, collect results. No coordination protocol needed. Swarms require persistent agents, role assignment, message passing, conflict resolution—massive overhead for marginal benefit at your scale.

## Part 7: Testing Philosophy

### The Core Problem

We've repeatedly encountered this pattern:

1. Write a skill with clear instructions
2. Verify the skill file exists and contains the instructions
3. Agent ignores the instructions anyway
4. Conclude "skill is broken" when actually "skill was never invoked"

This is the **existence vs behavior gap**. Testing "skill exists" is trivial. Testing "skill produces intended behavior" requires different techniques.

### Testing Pyramid for Agent Orchestration

```
                    ┌─────────────────┐
                    │   Behavioral    │  ← Hard, slow, high signal
                    │   Validation    │     (manual protocol)
                    └────────┬────────┘
                             │
               ┌─────────────┴─────────────┐
               │    Integration Tests      │  ← Medium difficulty
               │  (CLI + file assertions)  │     (bash scripts)
               └─────────────┬─────────────┘
                             │
          ┌──────────────────┴──────────────────┐
          │           Unit Tests                 │  ← Easy, fast
          │  (internal packages, property-based) │     (Go tests)
          └──────────────────────────────────────┘
```

### What Each Level Validates

| Level       | Tests                                               | Catches                          |
| ----------- | --------------------------------------------------- | -------------------------------- |
| Unit        | Functions work in isolation                         | Logic bugs, edge cases           |
| Integration | CLI commands produce expected files                 | Plumbing issues, format problems |
| Behavioral  | Agent actually follows instructions over N sessions | Skill invocation failures        |

### Key Principles

1. **Deterministic > Probabilistic** - `projctl` commands are deterministic; test them thoroughly. Agent behavior is probabilistic; use protocols, not assertions.

2. **Test the contract, not the content** - Don't test "SKILL.md contains X instruction." Test "when skill runs, X outcome occurs."

3. **Property-based for edge cases** - Use rapid/fast-check to generate random inputs. Catches edge cases humans miss.

4. **Manual protocols for behavioral** - Some things can't be automated. Document protocols, execute them, record results.

5. **Regression tests from corrections** - Every correction becomes a test case. "Agent did X wrong" → test that X is now right.

### Anti-Patterns to Avoid

| Anti-Pattern              | Why It Fails                                     |
| ------------------------- | ------------------------------------------------ |
| Testing file existence    | File can exist but never be invoked              |
| Testing file content      | Content can be correct but ignored               |
| Assuming skill invocation | 56% of skills are never invoked (Vercel finding) |
| Single manual test        | Probabilistic behavior needs repeated trials     |
| Testing in isolation      | Integration failures happen at boundaries        |

### Success Metrics

For each phase, we define concrete metrics:

| Phase                   | Metric                                               | Target             |
| ----------------------- | ---------------------------------------------------- | ------------------ |
| Structured Results      | All skills return valid result.toml with decisions   | 100%               |
| CLI Completeness        | `/project new` completes pm-interview phase          | 100%               |
| Model Routing           | Correct model in log for 10 skill dispatches         | 100%               |
| Cost Visibility         | Token totals match + budget alerts trigger           | 100%               |
| Learning Loop           | Correction rate drops after intervention             | >50%               |
| Parallel Dispatch       | 3 independent tasks complete in <1.5x single         | 100%               |
| Territory Mapping       | Token usage reduction in exploration phases          | >30%               |
| Graceful Degradation    | Recovery from simulated failures                     | 100%               |
